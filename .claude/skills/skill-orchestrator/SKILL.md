# Skill Orchestrator — 意図解析＆自動ルーティング

## 動作原則

1. **最小選定**: 効くスキルだけ選ぶ。数を増やしても品質は上がらない
2. **段階的開示**: 必要なスキルだけロード。全部同時に読まない
3. **プロジェクト固有スキル優先**: `.claude/skills/` 内のスキルを外部より優先
4. **自律実行**: 計画を出したら確認を待たず実行に移る

---

## PHASE 1: 意図解析（MANDATORY）

プロンプトを受け取ったら、まず以下4要素を内部で特定:

1. **WHY（本質的目的）** — このプロンプトが存在する根本的理由
2. **WHAT（ゴール状態）** — 完了時に何が存在しているべきか
3. **HOW（制約条件）** — ツール、言語、期限、フォーマット、環境
4. **WHERE（フェーズ）** — 企画 / 設計 / 実装 / 検証 / 運用

---

## PHASE 2: 目的類型の判定

### TYPE A: Create（ゼロから作る）
**シグナル**: 作る / 新規 / 構築 / 開発 / MVP / build / create / new
**実行構図**: 設計 → 実装（TDD並列）→ レビュー

### TYPE B: Improve（既存を改善する）
**シグナル**: 改善 / 最適化 / リファクタ / 高速化 / fix / optimize / refactor
**実行構図**: 調査 → 根本原因特定 → 修正 → 検証

### TYPE C: Secure（守りを固める）
**シグナル**: セキュリティ / 脆弱性 / 監査 / audit / security
**実行構図**: 脅威分析 → 脆弱性スキャン → 修正 → 再監査

### TYPE D: Ship / Grow（届ける・広める）
**シグナル**: ローンチ / デプロイ / 集客 / growth / launch / deploy
**実行構図**: ビルド検証 → デプロイ → 動作確認

### TYPE E: Test / Verify（検証する）
**シグナル**: テスト / 検証 / 確認 / test / verify / check
**実行構図**: 対象特定 → テスト実行 → 結果分析 → 修正

### TYPE F: Research / Analyze（調べる）
**シグナル**: 調査 / 分析 / 比較 / research / analyze
**実行構図**: 情報収集 → 構造化 → レポート

---

## PHASE 3: スキル選定

### 選定基準
1. **合計3-5個以内**に絞る
2. `.claude/skills/` 内のプロジェクト固有スキルを最優先
3. 同一領域の重複は排除

### Publish Gate固有スキル（優先ロード）

| スキル | 用途 |
|--------|------|
| `lp-analysis` | LP分析パイプライン（4ステップ）の実装・改善・デバッグ |
| `share-url-debug` | 共有URL・バイラルループのデバッグ |
| `security-audit` | SSRF防御・セキュリティ要件の監査 |
| `test-and-build` | テスト実行・ビルド検証 |
| `deploy-guide` | デプロイ手順・環境構築 |
| `chrome-extension` | Chrome拡張の開発・デバッグ |

### 汎用スキルパターン

| パターン | 適用場面 |
|----------|---------|
| 体系的デバッグ | エラー発生時 → 再現 → 切り分け → 修正 |
| TDD | 新機能実装時 → テスト先行 → 実装 → リファクタ |
| コードレビュー | 変更後 → 品質チェック → セキュリティ確認 |

---

## PHASE 4: 出力フォーマット

```
━━━ Skill Orchestrator ━━━
目的類型: TYPE X - [名称]

選定スキル:
1. `skill-name` — [役割] — [選定理由]
2. ...

実行構図:
skill-A → skill-B + skill-C（並列）→ skill-D
━━━━━━━━━━━━━━━━━━━━━
```

計画出力後、ユーザーの確認を待たず最初のスキルの実行に移行する。
