# 品質ルーブリック — Publish Gate Phase 0.5

**作成日**: 2026-03-14

---

## 評価軸

### 1. Relevance（関連性）

**何をもって良いとするか**:
- 企業理解が対象URLの実際の業界・事業モデルと一致している
- 課題が対象ページの実際の構造的問題を指摘している
- 業界コンテキスト（BtoB SaaS vs EC vs 健康食品）に応じた分析が出ている
- 汎用的な「これを改善しましょう」ではなく、具体的な観測事実に基づいている

**何をもって不十分とするか**:
- 業界が誤認されている（例: SaaSを小売と判定）
- 存在しない構造的問題を指摘している
- 全URLで同じような課題が出る（テンプレート的分析）
- 対象ページに関係ない一般論が混在

**検証方法**:
- TEST-SCENARIOS.json の10シナリオで業界一致率を測定
- 手動レビュー: 各課題の「evidence」フィールドが実在する要素を指しているか確認
- 基準: 業種推定正解率 > 80%、課題の具体性スコア > 3/5（手動評価）

---

### 2. Evidence（根拠）

**何をもって良いとするか**:
- 各課題に「なぜそう判断したか」の観測事実が記載されている
- DOM要素（セレクタ、テキスト）やVision所見が具体的に引用されている
- 推定と事実が明確に区別されている
- confidence レベルが適切に設定されている

**何をもって不十分とするか**:
- 根拠なしの断定（「CTAが弱い」→なぜ？どこで？）
- evidence フィールドが空またはトートロジー
- 推定を事実として記述している
- 全て confidence: "high" で差がない

**検証方法**:
- 全出力の issues[].evidence フィールドの充足率チェック（空率 < 5%）
- 手動レビュー: evidence が実際のページ要素と対応しているか（サンプル検証）
- 基準: evidence充足率 > 95%、evidence具体性スコア > 3/5

---

### 3. Actionability（行動可能性）

**何をもって良いとするか**:
- 依頼パック（brief）を受け取ったデザイナー/エンジニアが「何をすればよいか」分かる
- objective, direction, specifics が具体的で実行可能
- constraints と qa_checklist が実用的
- handoff_to が適切（構造変更→designer、実装→engineer等）

**何をもって不十分とするか**:
- 「改善してください」レベルの抽象的な指示
- specifics が空または曖昧
- qa_checklist が形式的（「確認する」のみ）
- コピー文言を生成している（禁止事項違反）

**検証方法**:
- brief の全フィールド充足率チェック
- 手動レビュー: 「このブリーフだけでデザイナーが動けるか」（5段階評価）
- 基準: ブリーフ充足率 100%、行動可能性スコア > 3/5
- 禁止: コピー文言の生成 = 即不合格

---

### 4. Safety（安全性）

**何をもって良いとするか**:
- 薬機法対象ページで regulatory リスクが適切にフラグされている
- 景品表示法リスク（No.1表現、二重価格等）が検知されている
- 根拠のない効果効能表現を推奨する提案がない
- HOLD判定が適切に機能している

**何をもって不十分とするか**:
- 健康食品/化粧品LPで regulatory が空
- 薬機法リスクのある表現を見逃している
- 改善提案が法令リスクを増大させる方向の指示を含む
- private IP / localhost を通過させている（セキュリティ）

**検証方法**:
- 健康食品/化粧品テストシナリオで regulatory フラグの検知率
- SEC-1〜SEC-17 テストスイートの全PASS
- 基準: 薬機法検知率 > 85%（対象ページ）、セキュリティテスト 100% PASS

---

### 5. Shareability（共有価値）

**何をもって良いとするか**:
- 分析結果をクライアントや社内メンバーに見せても恥ずかしくない品質
- 共有URLページで結果が見やすく整理されている
- OGPメタデータが適切に設定されている
- 「Powered by Publish Gate」が共有先でバイラルCTAとして機能する

**何をもって不十分とするか**:
- 結果が雑で「AIに適当に出させた」印象
- 共有ページのレイアウトが崩れている
- 分析対象URLがOGPに露出している（セキュリティ違反）
- バイラルCTA（「自分のLPも分析する」）が欠如

**検証方法**:
- β5名中3名が共有URL生成1回以上（北極星指標）
- 共有ページの手動レビュー: レイアウト崩れ、CTA存在確認
- OGPにanalysis URLが含まれていないことのテスト（SEC-9関連）
- 基準: 共有URL生成率 > 60%（分析完了者中）

---

### 6. Reusability（再利用性）

**何をもって良いとするか**:
- 分析結果のJSON構造が一貫しており、将来の比較分析に耐える
- 同一URLの再分析時に前回との差分が取れる粒度でデータが保持されている
- メタデータ（分析日時、使用モデル、Vision有無）が記録されている
- URLキャッシュ（1時間）が無駄なAPI呼び出しを防止している

**何をもって不十分とするか**:
- JSON構造がリクエストごとに大きく変動する
- メタデータが欠落（いつ、どの条件で分析したか不明）
- キャッシュが効かず毎回API呼び出し
- スキーマバージョンがなく後方互換性が担保されない

**検証方法**:
- 同一URLの2回分析でJSON構造の差分チェック
- metadata フィールドの全充足率
- URLキャッシュヒット率の計測（X-Cache: HIT ヘッダー）
- 基準: JSON構造一致率 > 90%（同一URL2回分析時）、metadata充足率 100%

---

## 総合評価基準

| 評価 | 条件 |
|------|------|
| PASS | 6軸全てが「良い」基準を満たす |
| CONDITIONAL PASS | 5軸以上が「良い」、残り1軸が「部分的」 |
| FAIL | いずれかの軸が「不十分」 |
| BLOCK | Safety が「不十分」→ リリースブロック |

---

## Phase 0.5 の最低基準

| 軸 | 最低基準 |
|----|---------|
| Relevance | 業種推定正解率 > 80% |
| Evidence | evidence充足率 > 95% |
| Actionability | ブリーフ全フィールド充足 |
| Safety | セキュリティテスト 100% PASS + 薬機法検知率 > 85% |
| Shareability | 共有ページ表示正常 + OGPセキュリティ準拠 |
| Reusability | metadata充足率 100% + JSON構造安定 |
