## 現設計の構造的矛盾7つ

ドキュメントの整合性や文章の完成度ではなく、**「このまま作って出したとき、ユーザーの前で壊れる箇所」**を指摘する。

---

### 矛盾1：AIがページの本文を読んでいない

Content Scriptのコードを見ると、抽出しているのはmeta情報、見出し(h1-h3)、CTA要素、フォーム、ナビゲーション、画像alt、構造化データ、統計値。

**ページの本文（body text）は抽出していない。**

プロンプト設計書にも「ページ本文は含まれません」と明記されている。つまりAIは「このページに何が書いてあるか」を知らずに改善提案を出している。

これは**「本を読まずに目次だけ見て書評を書く」のと同じ**。

例えば、LPのファーストビューに「業界シェアNo.1」と書いてあっても、AIには見えない。見出しに「実績」とあれば「実績セクションがある」ことはわかるが、「実績の見せ方が弱い」とか「数字が足りない」という提案は構造的に出せない。

**インタビューの七條が求めているのは「訴求の質」の判断。** 「ファーストビューのキービジュアルの中でのメッセージ」「顧客のニーズに刺さるナラティブ」を評価してほしい、と明確に言っている。本文が入力されないAIにはこれが原理的にできない。

**再設計案：本文を抽出する。ただし全文は不要。**

FV（上位1000px）のテキスト＋各CTAの周囲200文字＋見出し直下の1段落。これでコンテキストウィンドウへの負荷を抑えつつ、「何が書いてあるか」をAIに伝えられる。meta descriptionとh1-h3だけでは「骨格」しか見えないが、FVテキスト＋CTA周辺コピーが加われば「訴求の質」に踏み込める。

### 矛盾2：「公開判断」というフレーミングがユーザーの言葉と一致していない

プロダクト名は「Publish Gate」。設計書は「公開判断を支援する」と定義している。

しかしインタビューの3名全員が**「公開前チェック」という概念をピンと来ていない。**

- 浦田型：「公開前チェックってよくわかんないですね」（原文そのまま）
- 藤野型：「公開前チェックってよくわかんないですね」（同じ反応）
- 七條型：「あなたが出してくれるものだったとしても、それに対して、それがどういう観点でいいのかとか、どういう観点でベターなのか、どういう結果を出すために有効なのかが出てるんだったら」→ **チェックではなく「根拠つきの改善提案」を求めている**

ユーザーは「門番（Gate）」を求めていない。**「前に進むための推進力」**を求めている。

「Publish Gate」という名前は、開発者目線で機能を説明するには正確だが、ユーザーの頭の中にある課題（数字が動かない、何をすればいいかわからない）とは接続していない。

**再設計案：プロダクトの核を「公開判断」から「改善の推進」にずらす。**

名前は変えてもいいし変えなくてもいい（後から変えられる）。だが、Side Panelの最初のボタンが「🔍 このページをチェック」になっている時点で、ユーザーの頭の中は「チェック＝合否」になる。

代わりに「📈 改善ポイントを見つける」にすれば、期待値が「合否」ではなく「示唆」になる。この1語で、AIが推定を外したときのガッカリ度が下がる。「チェック」で間違えたら信用を失うが、「改善ポイントを探す」で精度が70%なら「7割も見つけてくれる」と受け取られる。

### 矛盾3：Layer 0の「おっ」が本当に出るか検証されていない

カスタマージャーニーで「30秒でFirst Value」と定義している。β検証計画で「Layer 0完走率 > 60%」をGO条件にしている。

しかし、Layer 0で出る分析結果の品質を左右する最大の変数は**プロンプトの精度**であり、プロンプトの精度を左右するのは**入力データの豊富さ**。

Layer 0の入力データはDOMの構造情報のみ（矛盾1の通り、本文すら入っていない）。この入力で「おっ」と思わせる出力が出るかは、**現時点で誰も確認していない**。

ロードマップのWeek 1で「代表5サイトで推定精度 > 70%」をT1.4の完了条件にしているが、「推定精度」の定義が曖昧。「ページの役割を正しく推定できたか」なのか「提案が実際に有効だったか」なのかで、70%の意味がまったく違う。

**最悪のシナリオ：** Layer 0で「このページはサービス紹介LPです。CVは問い合わせです」と推定は正しいが、提案が「CTAをもっと目立たせましょう」程度の汎用的なアドバイスしか出ない。ユーザーは「こんなのAIに聞かなくてもわかる」と思い、2回目のスキャンをしない。

**再設計案：プロンプト品質のベンチマークを開発着手前にやる。**

Week 1のT1.4にある「プロンプトv1」を、Week 0（=開発着手前の1日）で手動テストする。

やり方：自分でブラウザの開発者ツールから5サイトのDOM情報を手動コピーし、Claude APIに直接投げる。出力結果を3名のインタビュイー（または社内マーケ担当）に見せて「この提案、やりたいと思いますか？」と聞く。

この1日のテストで、Layer 0の入力データが十分かどうかが判明する。足りなければ、矛盾1の修正（本文抽出）が開発のDay 1から必要になる。足りていれば、自信を持ってWeek 1に入れる。

**これをやらずに3週間開発するのはリスクが高すぎる。**

### 矛盾4：「即時反映」と「実際のデプロイ」の間の崖

Side Panelで「即時反映」ボタンを押すと、ユーザーのブラウザ上でDOMが書き換わる。ユーザーは変更を「目で見る」ことができる。

だが、この変更は**ユーザーのブラウザのみに存在する**。ページをリロードすれば消える。他の誰にも見えない。本番環境には何の影響もない。

インタビューの藤野型は「実行って押すだけで、それが実装されてABテスト化されて、2週間後に結果が通知される」を理想として語っている。現設計の「即時反映」は、この理想の1%しか満たしていない。

現設計の「即時反映」は「プレビュー」であって「実行」ではない。だが、UIの表現（「即時反映」「適用」というラベル）はユーザーに「何かが起きる」期待を持たせてしまう。

**再設計案：「即時反映」の位置づけを明確にし、名前を変える。**

「即時反映」→「プレビューで確認」に名前を変更。ボタンの色もprimary（青）ではなくsecondary（グレー系）にする。

その代わり、「依頼パック」を最も目立つアクションにする。なぜなら、3名のインタビュイーのうち2名（浦田型・藤野型）にとっての最大価値は「デザイナー/開発者に渡せる指示書が自動生成される」ことだから。

```
現設計のProposals画面のアクション:
  [即時反映]  [依頼パック]  [保留]
   primary     secondary    text

再設計のProposals画面のアクション:
  [プレビュー]  [依頼パックを作る]  [メモを残す]
   secondary     primary             text
```

「即時反映」を殺す必要はない。ただし「これはプレビューです」と明示し、**主役を「依頼パック」にする**。依頼パックこそが、第3層ペイン（実装に渡すと壊れる）の直接的な解消であり、最も早く・最も確実にユーザーの時間を削減する機能。

### 矛盾5：「ログ蓄積で精度向上」が仕組みとして弱い

設計書のMOATは「判断ログの蓄積→提案精度の向上→使うほど良くなる複利構造」。β検証の仮説H2も「10回目の推定修正率が1回目より10%以上低下」。

しかし、実装を見ると、Layer 1で送信される過去ログは以下の構造：

```json
[
  { "judgment": "PASS", "reason": "...", "result": "effective/ineffective/pending" },
  ...（最大10件）
]
```

プロンプトには「このログから読み取れるパターン（勝ちパターン、失敗パターン）を提案に反映してください」と書いてある。

**問題：このデータで「パターン」を読み取れるほどの情報量がない。**

ログには「PASS」「理由」「結果」はあるが、**何をどう変えたか（Before/After）が入っていない**。AIは「この人は過去にPASSを3回出した」ことは知るが、「FVのキービジュアルに実績数値を追加したらCVR 120%になった」という**学習に必要な因果関係**を知ることができない。

これでは、10回使っても50回使っても、AIの提案品質は「DOM入力の質」で頭打ちになる。

**再設計案：ログに「変更内容」と「数値変化」を含める。**

Result画面の結果記録で、ユーザーが選んだ提案のBefore/After（AIが生成済み）を自動で結果ログに含める。

```json
{
  "page_url": "example.com/lp",
  "proposal_title": "FVに実績数値を追加",
  "category": "trust",
  "before": "会社紹介のみ",
  "after": "「導入企業300社・継続率98%」を追加",
  "result": "effective",
  "result_detail": "CTR +15%",
  "recorded_at": "2026-03-15"
}
```

こうすれば、AIが10件のログを見たとき「この会社では、trustカテゴリの改善（実績数値の追加）が効きやすい傾向がある」と推論できる。これは本当の意味での「学習」であり、MOATになる。

現設計のログは「日記」。再設計のログは「実験ノート」。実験ノートでなければ、次の実験の精度は上がらない。

### 矛盾6：価格体系とLayer設計のねじれ

価格設計書の構造：

```
Layer 0 (Free, アカウント不要)  → 5回/日, DOMのみ, ローカル50件
Layer 1 (Free, アカウントあり)  → 10回/日, DOM+ログ, サーバー保存
Pro ¥9,800/月                  → 無制限, DOM+ログ+GSC+GA4, 自動追跡, レポート
Team ¥49,800/月                → Pro全機能+共有+権限
```

**問題：Layer 0→1の差が小さすぎ、Layer 1→Proの差が大きすぎる。**

Layer 0→1：回数が5→10、ログがローカル→サーバー。これだけのためにアカウントを作る動機が弱い。「ログの永続保存」は、ログが10件溜まるまで価値を感じにくい。つまり**Layer 0を5日以上使い続けた人だけがLayer 1に来る**。

Layer 1→Pro：¥0→¥9,800のジャンプ。GSC連携・GA4連携・自動追跡・レポート・テンプレートが一気に入る。機能は豊富だが、「どの機能に¥9,800の価値があるのか」が曖昧。

インタビューで七條型は「¥5,000〜6,000で全然悪くない」と言っている。浦田型は「¥5,000超えたら会社経費」。¥9,800は許容範囲ギリギリであり、**「ちょっと試してみよう」の衝動買いラインではない。**

**再設計案：Layer 1に「無料だけど明確に嬉しい機能」を入れ、ProとBasicに分ける。**

```
Layer 0 (Free, アカウント不要)
  → 3回/日（5回は多い。3回で「もっと欲しい」を早く作る）
  → DOMのみ, ローカル保存

Layer 1 (Free, アカウントあり)
  → 10回/日
  → サーバーログ＋「依頼パック」が使い放題 ← ここが核
  → 過去ログ参照によるAI精度向上

Basic ¥4,980/月 ← 新設
  → 無制限スキャン
  → GSC連携（検索クエリベースの分析）
  → 承認メモ生成
  → レポート生成（基本版）

Pro ¥14,800/月 ← 値上げ
  → Basic全機能
  → GA4連携（CVR・行動データ）
  → 自動追跡（2週間後に結果通知）
  → テンプレート管理
  → 優先サポート

Team ¥49,800/月（5名、追加¥8,000/名）
  → Pro全機能＋共有ログ＋権限管理
```

**ポイント：**

- Layer 0→1の転換動機は「依頼パック」。インタビューで浦田型が最も強く反応した機能。アカウントを作る理由として「ログの永続保存」より遥かに強い
- ¥4,980は「個人でも払える、会社なら即決」のライン。七條型が言った「¥5,000〜6,000で悪くない」にぴったり
- ¥14,800は「GA4連携＋自動追跡」の本気層向け。藤野型が「いくらでも払う」と言った機能がここに入る
- Layer 0の回数を5→3に減らすのは、「物足りなさ」を早く感じさせるため。3回 = 主要3ページで1日分。4ページ目でアカウント作成の動機が生まれる

### 矛盾7：ターゲット顧客の「誰を最初に獲るか」が曖昧

3名のペルソナは本質的に違うユーザーで、求めるものが異なる：

| | 七條型（責任者） | 浦田型（担当者） | 藤野型（代理業） |
|---|---|---|---|
| 最大の痛み | チームに施策の考え方を教える時間 | 数字が悪い理由がわからない | クライアント報告の手間 |
| 「おっ」の瞬間 | 分析の自動化 | 依頼パック | クライアント提案の即時完成 |
| 課金判断 | 即決（部門予算） | 上申必要（1-2週間） | 即決（個人経費） |
| 求める価値 | チーム標準化 | 個人の生産性 | 案件あたりの効率化 |
| 拡大パス | Team課金 | 社内横展開 | 案件数の増加 |

**3つ全てを同時に満たそうとすると、どれも中途半端になる。**

β検証計画のWave 0は社内（七條型）で始めるが、七條型が最も価値を感じるのは「チーム展開＋GSC/GA4連携」であり、これはLayer 2-3の機能。Layer 0でのドッグフーディングでは、七條型の本当の「おっ」は出ない。

一方、**Layer 0で最も「おっ」が出やすいのは藤野型**。なぜなら：
- 藤野型の「おっ」は「クライアントのページでAI提案が出る→即座にスクショを撮ってSlackで送れる」
- これはLayer 0の機能範囲内で完結する
- 藤野型は課金判断が即座（個人経費）
- 藤野型のXでの発信が次のユーザー獲得に直結する

**再設計案：最初のβを藤野型（代理業・フリーランス）で始める。**

理由：
1. Layer 0だけで価値が完結する
2. 課金までのリードタイムが最短（即決）
3. 口コミ効果が最も高い（Xでの発信＝次の藤野型を連れてくる）
4. 1人あたりのLTV期待値が高い（案件数 × 月額）

社内でのドッグフーディング（七條型テスト）は並行して行うが、「GO/NO-GO判定の対象」は藤野型にする。

Wave設計の変更：

```
Wave 0（Week 4-5）：藤野型 3-5名
  → 代理業/フリーランスのマーケ支援者
  → 自分のクライアントのページで使ってもらう
  → 判定：「クライアントにそのまま見せられる品質か」

Wave 0.5（Week 4-6）：七條型 1名（Hashigaya Takuya内）
  → 並行ドッグフーディング
  → Layer 1-2の機能要件の精緻化

Wave 1（Week 7-8）：七條型の既存顧客10社
  → Layer 0-1提供

Wave 2（Week 9-10）：藤野型 × 公募50名
  → Layer 0-2提供
```

## ゼロから再設計した場合の全体像

上記7つの矛盾を踏まえ、ゼロから設計し直す。

### 設計思想：「ページを開くだけで、改善の判断が前に進む」は正しい。ただし、中身を変える。

| 要素 | 現設計 | 再設計 |
|---|---|---|
| AIの入力 | DOMの構造情報のみ | 構造＋FVテキスト＋CTA周辺コピー＋見出し下テキスト |
| 核の機能 | 即時反映（プレビュー） | 依頼パック（指示書自動生成） |
| フレーミング | 公開判断（合否） | 改善推進（示唆＋次のアクション） |
| ログの中身 | 判断種別＋理由＋結果 | 判断＋Before/After＋数値変化（実験ノート） |
| 価格の刻み | Free→¥9,800の崖 | Free→¥4,980→¥14,800の階段 |
| 最初のターゲット | 社内 | 代理業/フリーランス |
| Layer 0のスキャン回数 | 5回/日 | 3回/日 |
| Layer 1の転換動機 | ログの永続保存 | 依頼パックが使い放題 |

### 再設計したLayer構成

```
Layer 0：「試す」（アカウント不要）
  3回/日 → ページ分析＋提案3件＋プレビュー
  ログはローカル20件
  依頼パックは「サンプル表示のみ」（コピー不可）← 転換動機

Layer 1：「使う」（無料アカウント）
  10回/日 → 分析＋提案＋プレビュー＋依頼パック＋承認メモ
  ログはサーバー無制限、過去ログによるAI精度向上
  結果記録（Before/After＋数値変化を自動含有）

Basic ¥4,980/月：「仕事にする」
  無制限スキャン＋GSC連携
  レポート生成（社内用/クライアント用）
  テンプレート管理

Pro ¥14,800/月：「武器にする」
  Basic全機能＋GA4連携＋自動追跡＋結果通知
  優先サポート

Team ¥49,800/月：「組織にする」
  Pro全機能＋5名＋共有ログ＋権限管理
```

**Layer 0→1の転換キーは「依頼パック」。**

Layer 0では提案カードの下に依頼パックの「プレビュー」が見える。「この品質の指示書がワンクリックで作れます」と見せる。ただしコピーボタンは無効で「アカウント作成でコピー可能」と表示。

これは遮断ではない。分析結果は全て見えている。提案も見えている。ただ、「次の行動（デザイナーに渡す）」に進むためにアカウントが必要。行動を遮断するのではなく、**行動の最後の1ステップ**だけをゲートにする。

### 再設計したContent Script（本文抽出の追加）

```javascript
// 追加：FV（上位1000px）のテキスト抽出
function extractFVText() {
  const elements = document.querySelectorAll('h1, h2, p, span, li, a');
  const fvTexts = [];
  for (const el of elements) {
    const rect = el.getBoundingClientRect();
    if (rect.top < 1000 && rect.top >= 0 && el.textContent.trim().length > 10) {
      fvTexts.push(mask(el.textContent.trim().substring(0, 200)));
    }
    if (fvTexts.length >= 30) break;
  }
  return fvTexts;
}

// 追加：CTA周辺のコピー（前後200文字）
function extractCTAContext(ctaElement) {
  const parent = ctaElement.closest('section, div, article') || ctaElement.parentElement;
  if (!parent) return '';
  return mask(parent.textContent.trim().substring(0, 400));
}
```

この追加により、AIの入力が以下に変わる：

```
現設計: meta + headings + CTAs(テキストのみ) + forms + navigation + images + stats
再設計: meta + headings + CTAs(テキスト+周辺コピー) + forms + navigation + images + stats + FVテキスト
```

トークン増加量は約500-1000トークン。API原価への影響は¥2-5/リクエスト。Layer 0の品質が劇的に上がることを考えれば、許容範囲。

## まとめ：何が一番大事か

7つの矛盾を書いたが、**最も致命的なのは矛盾1（AIがページの本文を読んでいない）と矛盾3（Layer 0の品質が未検証）**。

この2つが解決されていなければ、他の6つをどんなに磨いても「おっ」は出ない。「おっ」が出なければ2回目のスキャンは起きない。2回目が起きなければLayer 1もProも存在しない。

逆に、矛盾1を修正して（FVテキストとCTA周辺コピーを抽出する）、矛盾3を検証して（開発前にプロンプトをAPI直打ちで5サイトテストする）、Layer 0で本当に「おっ」が出ることを確認してから開発に入れば、残りの5つは走りながら修正できる。

**やるべき最初の1日：**

1. 手動で5サイトのDOM＋FVテキスト＋CTA周辺コピーを収集（2時間）
2. 再設計プロンプトでClaude APIに投げる（1時間）
3. 出力を社内マーケ担当に見せて「この提案、やりたいですか？」と聞く（1時間）
4. 3/5が「やりたい」と言えば開発GO。言わなければプロンプトを修正してもう1回

この1日をスキップして3週間開発するのと、この1日をやってから3週間開発するのでは、3週間後に手元にあるものの価値がまるで違う。
