# Publish Gate カスタマージャーニー

**版：v3.0　更新日：2026-03-08**

---

## 1. v3.0のジャーニー設計思想

**Webダッシュボードが本体。Chrome拡張はリテンション/リマーケ。**

v2.0はChrome拡張中心のジャーニーだったが、v3.0ではWebダッシュボードのURL入力から始まるジャーニーに全面改訂。成果物（共有URL、依頼書、CSV）が外に出ることでバイラルが回る設計。

---

## 2. 3つのペルソナ

**浦田型（主ターゲット）：事業会社マーケ担当**
「なんでその数字かわからない」が最大の痛み。URLを入れて「おっ」となり、共有URLを上司に送り、Starter検討に至る。上司の承認が必要。

**藤野型（初期バイラル起点）：代理業/フリーランス**
クライアントのLP分析→結果URLをクライアントに送る→クライアントが触る→フライホイール起動。「4案件回せるなら年収1億」。

**七條型（Business層）：マーケ責任者**
チームメンバーに使わせる→レポートが組織内を回る→Business検討。「インターン3人分」。部門予算で即決可能。

---

## 3. ジャーニー全体像

```
認知 → URL入力 → 「おっ」 → 共有URL生成 → 依頼書 → Starter → 広告訴求 → GSC/GA4 → Pro → チーム → Business
```

---

## 4. Phase別ジャーニー

### Phase 0.5：藤野型がURL入力→共有URLでバイラル起動

```
藤野: クライアントLP URL入力 → 分析結果（タブ1） → 「これ使える」 → 共有URL生成 → クライアントSlackに送付
クライアント: 共有URL開く → Read-Only結果 → 「自分もやりたい」 → Free登録 → 自分のURLで分析 → …
```

- センターピン：藤野型が結果URLをクライアントに送ったか
- 成功指標：共有URL生成率 > 30%、共有URL経由の新規登録 > 10%

### Phase 1：浦田型がURL入力→「おっ」→Starter→依頼書

```
浦田: 自社LP URL入力 → タブ1（LP分析） → 課題一覧 → 「おっ、これが原因か」
     → 依頼書生成 → デザイナーにSlack送付 → 「これで伝わる」
     → Starter検討 → 上司に共有URL送付 → 承認 → Starter課金
     → タブ2（広告訴求） → CSV → 広告チームへ
```

- センターピン：依頼書を外に送ったか
- 成功指標：依頼書生成率 > 20%、Starter転換率 > 5%

### Phase 2：浦田型がGSC/GA4連携→Pro→週次レポート

```
浦田: GSC/GA4連携 → タブ3-5が開放 → 市場・流入・競合が見える
     → Pro課金 → 週次レポート自動生成 → 上長にメール転送
     → 「もうこれなしで報告できない」
```

- センターピン：レポートを組織内転送したか
- 成功指標：Pro転換率 > 3%、週次レポート開封率 > 50%

### Phase 3：七條型がチーム導入→Business

```
七條: チームメンバーにアカウント配布 → 各自が担当サイトを分析
     → レポートが組織内を回る → 「全員のデフォルトツール」
     → Business検討 → タブ6（事業分析）開放
```

- センターピン：チーム全員のデフォルトになったか
- 成功指標：Business転換率、チーム内週次アクティブ率 > 60%

## 5. バイラルジャーニー（成果物が外に出る経路）

### 経路1：共有URL（最強）

```
分析結果 → 共有URL生成 → Slack/メール → 受け取り側が閲覧（アカウント不要）
→ 「自分のURLでもやりたい」 → Free登録 → 分析 → 共有URL生成 → …
```

### 経路2：依頼書Powered by

```
依頼書生成 → デザイナー/エンジニアが受領 → フッターに「Powered by Publish Gate」
→ 「何このツール？」 → Free試用 → …
```

### 経路3：CSV→広告チーム

```
タブ2（広告訴求） → CSV出力 → 広告運用チームが利用
→ 「この訴求文どこから？」 → Publish Gate認知 → 導入検討
```

---

## 6. 「おっ」の瞬間（ペルソナ別）

| ペルソナ | 「おっ」の正体 | 所要時間 |
|---------|-------------|---------|
| 浦田型 | 「数字が悪い理由がわかった」（課題一覧の明快さ） | URL入力から30秒 |
| 藤野型 | 「クライアントへの提案がこの場で完成する」（共有URL） | URL入力から60秒 |
| 七條型 | 「これをインターンに使わせればいい」（依頼書の品質） | 2回目の利用時 |

---

## 7. 離脱ポイントと対策

| 離脱ポイント | 離脱理由 | 対策 |
|---|---|---|
| URL入力前 | 何ができるか不明 | ダッシュボードトップにサンプル分析結果を表示 |
| 初回分析結果 | 推定が的外れ | Vision API + 4ステップパイプラインで精度確保 |
| 共有URL生成前 | 共有する動機がない | 結果ページの目立つ位置に共有ボタン配置 |
| Free→Starter | 上司の承認が得られない | 共有URLで上司に価値を見せる導線 |
| Starter→Pro | GSC/GA4連携が面倒 | OAuth + 「読み取りのみ」明記 |
