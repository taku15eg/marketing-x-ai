### Phase B: ゼロベース再設計（2026-02-28後半）

#### Session 9: 「ゼロから再設計するなら？」
**日時：** 2026-02-28 12:26〜
**ユーザーリクエスト：** 「いまプロジェクトをゼロから自由に再設計できるとしたら、どういう設計にしますか？」
**Claude提案（第1案）：** 「scan→suggest」から「learning loop-first」へのアーキテクチャ転換。Web App＋データ統合を軸にする案
**構造的矛盾7つの指摘：**

1. **AIがページ本文を読んでいない** — content-scriptがmeta/h1-h3/CTA/フォームのみ抽出。本文テキスト未取得。「本を読まずに目次だけ見て書評を書く」状態
2. **「公開判断」フレーミングのミスマッチ** — インタビュー3名全員が「公開前チェック」にピンと来ていない。ユーザーは「門番」でなく「推進力」を求めている
3. **PASS/FAILが精度70%で成立しない** — 合否判定は高精度必須。改善提案なら精度70%でも価値がある
4. **即時反映が核ではない** — 最もペインが深いのは「人に渡す」工程（ハンドオフパック）
5. **7画面フローが360px Side Panelに収まらない** — 画面数の圧縮が必要
6. **ログ構造がMOATにならない** — 判定＋理由ではなく、Before/After＋結果のラボノート形式が必要
7. **課金ゲートの位置が最適でない** — ログ永続化ではなく「依頼パックのコピー」に置くべき

#### Session 11: Progressive Value再設計（決定版）
**日時：** 2026-02-28 15:09〜
**ユーザーの反論と追加リクエスト：**
- 「でも、サイトにするとアクセスが減らない？拡張機能のほうが簡単にアクセスできそう」
- 「サードパーティ連携は必須？アカウント作成は？」

- 「全ファイルをどう更新するか？すべて走れる状態まで進めて」

**決定された設計思想：** Progressive Value Architecture（Layer 0→1→2→3）
- **Layer 0（即体験）：** インストール→1クリックで3提案。アカウント不要、連携不要
- **Layer 1（習慣化）：** アカウント作成→ログ蓄積→精度向上
- **Layer 2（データ強化）：** GSC連携→クエリベース分析
- **Layer 3（フル機能）：** GA4連携→自動トラッキング→レポート

**ユーザー満足点：** 「価値体験ファースト」の設計哲学が採用された。連携なしで最初の価値を感じられるUX

#### Session 12: Web vs 拡張機能の戦略議論
**日時：** 2026-02-28 15:18〜
**ユーザーの問い：** 「拡張機能のみだとインストール必要。Webペライチのほうがアクセス多いのでは？MVPとグロース両方取るならどっちがいい？私の指摘が間違っていれば正直教えて欲しい」

## 4. 重要な設計意思決定の記録

### 4.1 フレーミングの転換
**Before（v0.x）：** 「公開判断」→ PASS/FAIL/HOLD
**After（v2.0）：** 「改善ポテンシャル」→ 3提案＋根拠＋ハンドオフ

**根拠：** インタビュー3名全員が「公開前チェック」にピンと来ていない。精度70%のAIに合否判定させると信頼が壊れる。改善提案なら精度70%でも「参考になる」で成立する。

### 4.2 AI入力データの拡張
**Before：** meta情報＋h1-h3＋CTA＋フォーム＋JSON-LD
**After：** 上記＋FVテキスト（上位1000px）＋CTA周辺200文字＋見出し直下1段落

**根拠：** 七條型が求める「訴求の質」判断は、本文を読まないAIには原理的に不可能。

### 4.3 画面構成の圧縮
**Before：** 7画面フロー
**After：** 3画面（Scan/Insight/Log）＋ボトムナビゲーション

**根拠：** 360px Side Panelでは画面遷移が多いほど離脱する。

### 4.4 課金ゲートの位置変更
**Before：** ログ永続化で課金
**After：** 「依頼パックのコピー」で課金

**根拠：** 最も知覚価値が高い瞬間（＝改善案を手に入れてチームに渡したい瞬間）にゲートを置く。

### 4.5 ログをラボノート形式に
**Before：** 判定＋理由の記録
**After：** Before/After＋結果の実験ノートブック

**根拠：** Ahrefs的MOATを構築するには、データ蓄積が精度向上につながる構造が必要。実験ノートは再利用可能で学習ループを形成する。

### 4.6 ハイブリッドWeb＋拡張機能
**ユーザーの懸念：** Webだけだとアクセスが減る。拡張機能のほうがワンクリック。
**決定：** WebはSEOファネル入口（URL入力→即結果→拡張機能インストール誘導）、拡張機能は日常ツール。
**根拠：** MVPの初期リーチとグロース段階の日常利用の両立。

## 5. ユーザーの核心的リクエストと満足ポイント

### 5.1 一貫して繰り返されたリクエスト
1. **「添付とすべてを理解のうえ」** — 表面的な理解でなく、全資料を精読した上での本質的分析を求めている
2. **「本質的な目的とゴール状態に沿って」** — 施策飛びではなく、常にゴール状態から逆算した設計
3. **「すべて走れる状態まで進めて」** — 途中で止めず、実行可能な完成度まで自律的に走り切ること
4. **「全て終わった？」** — 完成度の確認。抜け漏れがないかの検証を重視

### 5.2 満足が確認されたポイント
1. **Progressive Value設計の採用** — 「まずは価値を体験してもらうUXにできないかな？」→ Layer 0の即体験設計で応答
2. **構造的矛盾の指摘** — 「ゼロから再設計するなら？」への7つの矛盾指摘は、単なるリファクタではなく「このまま作って出したときにユーザーの前で壊れる箇所」を明示
3. **インタビュー原文への忠実さ** — 抽象化したペルソナではなく、七條型/浦田型/藤野型として実際の発言に基づくジャーニー設計
4. **正直なフィードバック** — 「私の指摘が間違っていれば正直教えて欲しい」→ Web onlyの弱点も拡張機能onlyの弱点も率直に分析した上でハイブリッド提案
5. **ZIP完全パッケージ** — 53ファイルを1つのZIPに格納し、即ハンドオフ可能な状態で提供

### 5.3 ユーザーが重視した原則
- **アップサイドフレーミング > 判定フレーミング** — PASS/FAILではなく「改善ポテンシャル」
- **ワークフロー埋め込み** — DeepLのCtrl+C+Cのように日常に溶け込むSide Panel
- **品質集中 > 機能幅** — DeepL/Ahrefsの教訓：集中した品質優位がMOATになる
- **データ蓄積によるMOAT** — ログがラボノートとして蓄積→精度向上→離脱コスト上昇
- **GO/NO-GOゲート** — 定量的判断基準で「弱い基盤の上に積まない」
