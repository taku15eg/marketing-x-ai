**0. Executive Summary**

4セッションのディスカッションから抽出された20項目の論点と、7つの構造的矛盾、そして全プロジェクトナレッジを横断分析した結果、現設計には「診断された7つの矛盾」に加えて**も8つ目の本質的な問題**がある。

それは「プロダクトの定義が2つ存在する」という問題だ。統合ドキュメントv1.2は「ページ単位の意思決定と実装運用を標準化するOS」と定義し、Deep
Research報告書は「Decision Ops
OS」と呼ぶ。一方、v2.0要件定義書は「ページを開いて数分で改善の判断が前に進むChrome拡張」と定義している。前者は「プラットフォーム」であり、後者は「ツール」である。

**このドキュメントでは、**「ツールから始めて、OSに育てる」という第3の道を提案する。

MVPは「1ページを開いて、改善の次の一手が前に進む」Chrome拡張。ただし、ログ構造・テンプレート構造・API設計は「将来OSになる」前提で設計する。これにより、MVPのスコープを小さく保ちつつ、拡張性を犠牲にしない。

**1. 本質的な目的（再定義）**

戦略分析サマリーの定義は正しいが、MVPには重すぎる。再定義する：

> *「ページを開いたときに、『何が悪いか』『なぜ悪いか』『どう直すか』が揃い、そのまま実装者に渡せる形で出力され、結果が次の判断に活きる。このサイクルが、1人でも回り続ける状態をつくる。」*

**2. 8つ目の本質的な問題：プロダクト定義の二重性**

既存の7つの矛盾は正確な診断だが、その上位にもう1つある。

**ドキュメント群の中で、「Decision Ops
OS」と「Chrome拡張ツール」という2つのプロダクト定義が並行している。**
統合ドキュメントv1.2とDeep
Research報告書は「ページ単位の意思決定運用を標準化するOS」と描き、5種類のユーザー像（SEO担当・CRO担当・PR担当等）を対象とする。v2.0要件定義書は「3名のインタビューから逆算したChrome拡張」と描く。

この二重性が、スコープの肌福れ（UI設計書の7画面遷移）、ターゲットの曖昧さ（F-1）、価格設計のねじれ（E-1/E-2）の根本原因。

**3. ゼロから再設計したコアアーキテクチャ**

**3.1 AI入力の再設計（矛盾A-1の解消）**

現設計の致命的欠陥は「AIがページの本文を読んでいない」こと。「本を読まずに目次だけ見て書評を書く」状態を解消する。

  --------------------------------------------------------------------------
  **入力データ**             **現設計**            **再設計**
  -------------------------- --------------------- -------------------------
  メタ情報                   ○                     ○

  見出し (h1-h3)             ○                     ○

  CTA要素                    テキストのみ          テキスト + 周辺200文字

  フォーム構造               ○                     ○

  FVテキスト（上位1000px）   ✖                     ○ 最大30要素

  見出し直下の1段落          ✖                     ○

  画像alt                    ○                     ○

  トークン増加量             ---                   +500〜1000 (¥2-5/req)
  --------------------------------------------------------------------------

トークン増加は¥2-5/リクエスト。Layer
0の品質が劇的に上がることを考えれば、完全に許容範囲。

**3.2 フレーミングの転換（矛盾B-1/B-2の解消）**

  ------------------------------------------------------------------------------------------------------
  **要素**         **現設計**               **再設計**                   **理由**
  ---------------- ------------------------ ---------------------------- -------------------------------
  プロダクト名     Publish Gate             Publish Gate（変更不要）     名前は後から変えられる

  メインCTA        🔍 このページをチェック  📈 改善ポイントを見つける    「チェック」→合否が期待される

  結果のフレーム   PASS / FAIL              改善提案×3件（優先度付き）   合否は行動を阻害する

  主役アクション   即時反映（プレビュー）   依頼パック（指示書生成）     第3層ペインの直接解消
  ------------------------------------------------------------------------------------------------------

「チェック」で間違えたら信用を失うが、「改善ポイントを探す」で精度が70%なら「7割も見つけてくれる」と受け取られる。この1語の差が、推定精度が不十分なMVPe期に決定的に重要。

**転換設計の核心：「行動の最後の1ステップ」だけをゲートにする**

Layer
0では分析結果も提案も全て見える。依頼パックの「プレビュー」も見える。ただしコピーボタンが無効で「アカウント作成でコピー可能」と表示。これは遮断ではなく、「デザイナーに渡す」という次の行動の最後の1ステップだけがゲート。

Layer
0のスキャン回数を5→3に減らすのは、「物足りなさ」を早く感じさせるため。3回
= 主要3ページで4ページ目でアカウント作成動機が生まれる。

**5. ログ構造の再設計：「日記」から「実験ノート」へ**

**5.1 現設計の問題（矛盾D-1/D-2）**

現設計のログには「judgment」「reason」「result」しかなく、「何をどう変えたか（Before/After）」が入っていない。これでは10回使っても50回使っても、AIの提案品質は「DOM入力の質」で頭打ちになる。

**5.2 再設計：実験ノート型ログ**

各ログエントリに以下を自動含有：

  -----------------------------------------------------------------------------
  **フィールド**   **ソース**                **用途**
  ---------------- ------------------------- ----------------------------------
  page_url         自動                      ドメイン単位の傾向分析

  proposal_title   AI生成済み                提案の分類

  category         AI生成済み                trust / cta / copy / structure等

  before           AI生成済み                変更前の状態

  after            AI生成済み                変更後の状態

  result           ユーザー入力（2タップ）   効果あり / 変化なし / 悪化

  metrics_delta    ユーザー入力（任意）      CVR/CTRの変化値

  selector         AI生成済み                対象セSSセレクタ
  -----------------------------------------------------------------------------

ユーザーが入力するのは「効果あり/なし/悪化」 +
「数値（任意）」の2タップだけ。Before/Afterと提案内容はAIが生成済みなので自動添付。

このログが10件溜まれば、AIは「この会社ではtrustカテゴリの改善が効きやすい。特にFVに数値を入れるパターンが2回成功している」と推論できる。これが本当の意味でのMOAT。

  LTV期待値                高（Team課金）         中（Basic課金）        高（案件数×月額）

  「おっ」の出やすさ       △                      ○                      ◎
  ---------------------------------------------------------------------------------------------

藤野型は(1) Layer 0だけで価値が完結、(2) 課金が即日、(3)
Xでの発信が次の藤野型を連れてくる、(4)
LTVが高い。SEVENDEX社内は並行ドッグフーディング。

**7. Phase 0：開発着手前の1日テスト（最重要）**

**これが全体のGO/NO-GO。**この1日をスキップして3週間開発するのと、この1日をやってから3週間開発するのでは、3週間後に手元にあるものの価値がまるで違う。

  ------------------------------------------------------------------------------------------------------------
  **時間**   **タスク**                                                         **目的**
  ---------- ------------------------------------------------------------------ ------------------------------
  2時間      5サイトのDOM + FVテキスト + CTA周辺コピーを手動収集                再設計した入力データでテスト

  1時間      再設計プロンプトでClaude APIに直打ち                               出力品質の検証

  1時間      出力を社内マーケ担当に見せて「この提案、やりたいですか？」と聞く   「おっ」が出るかの検証
  ------------------------------------------------------------------------------------------------------------

**GO条件：3/5が「やりたい」と言えば開発着手。言えなければプロンプトを修正してもう1回。**
