**1. 事前準備（所要: 30分）**

**1-1. 環境準備**

+-----------------------------------------------------------------------+
| **STEP 1-A: Anthropic APIキーを取得**                                 |
|                                                                       |
| 1\. https://console.anthropic.com にアクセス                          |
|                                                                       |
| 2\. Settings → API Keys → Create Key                                  |
|                                                                       |
| 3\. キーをコピーして安全な場所に保存                                  |
|                                                                       |
| ※ 既にキーがある場合はスキップ。テスト全体で約¥50-100のAPI費用。      |
+-----------------------------------------------------------------------+

+-----------------------------------------------------------------------+
| **STEP 1-B: テストスクリプトの準備**                                  |
|                                                                       |
| 1\. publish-gate-v3.tar.gz を展開                                     |
|                                                                       |
| 2\. ターミナルで展開したディレクトリに移動                            |
|                                                                       |
| 3\. 以下のコマンドでスクリプトの動作を確認:                           |
+-----------------------------------------------------------------------+

+-----------------------------------------------------------------------+
| cd publish-gate-v3                                                    |
|                                                                       |
| node \--check test/phase0-prompt-test.mjs                             |
|                                                                       |
| \# → エラーが出なければOK                                             |
+-----------------------------------------------------------------------+

**1-2. テスト対象ページの選定**

Phase
0の信頼性は「どのページでテストするか」で決まる。以下の基準でクライアント
or 自社のページを5件選ぶ。

  -------- ----------------------- ---------------------------------- --------------------------
  **\#**   **ページ種別**          **選定基準**                       **例**

  1        獲得LP                  BtoB SaaS or                       自社サービスのLP
                                   サービスの新規獲得ページ           

  2        サービスページ          制作会社・コンサルのサービス紹介   SEVENDEX自社ページ

  3        コーポレートサイトTOP   会社紹介のメインページ             クライアントの企業サイト

  4        料金ページ              SaaS or スクールの料金表ページ     クライアントの料金ページ

  5        事例ページ              導入事例・実績紹介                 クライアントの事例集
  -------- ----------------------- ---------------------------------- --------------------------

+-----------------------------------------------------------------------+
| **💡 ページ選定のコツ**                                               |
|                                                                       |
| ・「改善の余地が                                                      |
| ある」ページを意図的に選ぶ（完璧なページだとAIの提案価値が測れない）  |
|                                                                       |
| ・                                                                    |
| 藤野型ユーザーが実際にクライアントから受け取るページに近いものを選ぶ  |
|                                                                       |
| ・最低3ページはSEVENDEX外のサイトを入れる（自社バイアス回避）         |
+-----------------------------------------------------------------------+

**2. ページデータ抽出（所要: 1-2時間）**

v3.0のcontent-script.jsが実際に取得するデータと同等の情報を、手動でChromeから抽出する。

**2-1. 方法A: スクリプト貼り付け（推奨）**

content-script.jsを直接ブラウザで実行する方法。最も正確。

+-----------------------------------------------------------------------+
| **STEP 2-A1: 対象ページをChromeで開く**                               |
|                                                                       |
| 1\. テスト対象のURLをChromeで開く                                     |
|                                                                       |
| 2\. ページが完全に読み込まれるまで待つ（ローディングが終わるまで）    |
+-----------------------------------------------------------------------+

+-----------------------------------------------------------------------+
| **STEP 2-A2: DevToolsを開いてスクリプトを実行**                       |
|                                                                       |
| 1\. F12（またはCmd+Option+I）でDevToolsを開く                         |
|                                                                       |
| 2\. Console タブをクリック                                            |
|                                                                       |
| 3\. content/content-script.js の中身を全文コピーしてConsoleに貼り付け |
|                                                                       |
| 4\. Enterで実行                                                       |
+-----------------------------------------------------------------------+

+-----------------------------------------------------------------------+
| **STEP 2-A3: データを取得してコピー**                                 |
|                                                                       |
| 1\. Consoleに以下を入力してEnter:                                     |
+-----------------------------------------------------------------------+

  -----------------------------------------------------------------------
  copy(JSON.stringify(window.\_\_publishGate.extractFeatures(), null, 2))

  -----------------------------------------------------------------------

これでクリップボードにJSON形式のページデータがコピーされる。

+-----------------------------------------------------------------------+
| **STEP 2-A4: JSONファイルとして保存**                                 |
|                                                                       |
| 1\. テキストエディタを開く（VSCode等）                                |
|                                                                       |
| 2\. 貼り付け（Ctrl+V / Cmd+V）                                        |
|                                                                       |
| 3\. page-1-lp.json のようなファイル名で保存                           |
|                                                                       |
| 4\. 5ページ分すべて繰り返す                                           |
+-----------------------------------------------------------------------+

**2-2. 方法B: サンプルデータで即テスト（時間がない場合）**

test/phase0-prompt-test.mjs にBtoB SaaS
LPとコーポレートサイトの2件のサンプルデータが組み込み済み。まずはこれで動かして雰囲気を掴み、その後で実データに差し替える方法。

  -----------------------------------------------------------------------------------
  **⚠️
  サンプルデータだけでGO/NO-GO判定はしない。必ず実データ3件以上でも検証すること。**

  -----------------------------------------------------------------------------------

**3. テスト実行（所要: 30分-1時間）**

**3-1. サンプルデータでの実行（まずこれをやる）**

+-----------------------------------------------------------------------+
| **STEP 3-A: テストスクリプトを実行**                                  |
|                                                                       |
| 1\. ターミナルで publish-gate-v3/ ディレクトリに移動                  |
|                                                                       |
| 2\. 以下のコマンドを実行:                                             |
+-----------------------------------------------------------------------+

+-----------------------------------------------------------------------+
| ANTHROPIC_API_KEY=sk-ant-xxxxx node test/phase0-prompt-test.mjs       |
|                                                                       |
| \# Windowsの場合:                                                     |
|                                                                       |
| set ANTHROPIC_API_KEY=sk-ant-xxxxx && node                            |
| test/phase0-prompt-test.mjs                                           |
+-----------------------------------------------------------------------+

実行すると、各ページについてAIの分析結果がJSON形式で出力され、自動品質スコアが表示される。

**3-2. 実データでの実行**

+-----------------------------------------------------------------------+
| **STEP 3-B: テストスクリプトに実データを追加**                        |
|                                                                       |
| 1\. test/phase0-prompt-test.mjs をエディタで開く                      |
|                                                                       |
| 2\. SAMPLE_PAGES 配列に、2-Aで保存したJSONを追加:                     |
+-----------------------------------------------------------------------+

+-----------------------------------------------------------------------+
| // SAMPLE_PAGES 配列に追加                                            |
|                                                                       |
| {                                                                     |
|                                                                       |
| name: \"クライアントA社 LP\",                                         |
|                                                                       |
| data: JSON.parse(fs.readFileSync(\"page-1-lp.json\", \"utf-8\")),     |
|                                                                       |
| },                                                                    |
+-----------------------------------------------------------------------+

ファイル先頭に const fs = require(\"fs\"); を追加するのを忘れずに。

**3-3. 出力の読み方**

  ------------------------- ---------------------------------- ------------------------------
  **出力項目**              **何を見るか**                     **良い兆候**

  page_estimation           ページ種別・CV・ターゲットの推定   実際と一致 or 近い

  proposals\[0\].title      最優先提案のタイトル               具体的で実行可能

  proposals\[0\].before     現状の引用                         実際のページ内容と一致

  proposals\[0\].after      変更後の具体例                     「これやりたい」と思える品質

  proposals\[0\].evidence   提案の根拠                         納得できる論拠がある

  自動スコア                構造的品質（%）                    80%以上
  ------------------------- ---------------------------------- ------------------------------

**4. レビュー（所要: 1時間）**

テスト結果をプリントアウト（またはSlackに貼り付け）して、社内マーケ担当に見せる。最低1名、理想は3名。

**4-1. レビュアーの条件**

  ------------ --------------------------- --------------------------------------
  **優先度**   **人物像**                  **理由**

  最優先       SEVENDEXマーケ責任者        最も日常的にページ改善を行っている人

  次点         クライアントワーク担当者    藤野型ユーザーに最も近い視点

  補助         他のマーケ担当者            多角的なフィードバック
  ------------ --------------------------- --------------------------------------

**4-2. レビューの進め方（1ページあたり10分）**

+-----------------------------------------------------------------------+
| **STEP 4-A: 文脈を共有する（1分）**                                   |
|                                                                       |
| 「このページに対して、AIが改善提案を3つ出しました。                   |
|                                                                       |
| 　あなたがこのページの担当者だとして、この提案を見てください」        |
|                                                                       |
| ※ Publish Gateの説明はしない。純粋に提案の品質だけを評価してもらう。  |
+-----------------------------------------------------------------------+

+-----------------------------------------------------------------------+
| **STEP 4-B: 各提案を1つずつ見せる（各3分）**                          |
|                                                                       |
| 以下の情報を順番に見せる:                                             |
|                                                                       |
| ① 提案タイトル                                                        |
|                                                                       |
| ② Before（現状）                                                      |
|                                                                       |
| ③ After（変更後）                                                     |
|                                                                       |
| ④ 根拠                                                                |
|                                                                       |
| 見せた後、1つだけ質問する:                                            |
+-----------------------------------------------------------------------+

  -----------------------------------------------------------------------
  **「この提案、やりたいですか？」**

  -----------------------------------------------------------------------

+-----------------------------------------------------------------------+
| **STEP 4-C: 回答を記録する**                                          |
|                                                                       |
| 回答は3段階で記録:                                                    |
|                                                                       |
| ◎ やりたい --- 「これは実際にやる価値がある」                         |
|                                                                       |
| △ 微妙 --- 「方向性はわかるが、このままでは使えない」                 |
|                                                                       |
| ✗ やらない --- 「的外れ」「当たり前すぎる」「具体性が足りない」       |
|                                                                       |
| 「微                                                                  |
| 妙」の場合、何が足りないかを必ず聞く（これがプロンプト改善のヒント）  |
+-----------------------------------------------------------------------+

**4-3. 追加の確認質問（全提案を見た後）**

全3提案を見終わった後、以下を聞く:

  ----------------------------------------------------------------- -----------------------------------
  **質問**                                                          **何を測っているか**

  「クライアントにこのまま見せられる品質ですか？」                  藤野型の「依頼パック」品質基準

  「AIが出したと聞いて驚きますか？それとも予想通りですか？」        「おっ」の有無

  「足りない情報は何ですか？（もっとこういうデータがあれば...）」   Layer 1-2で追加すべき入力
  ----------------------------------------------------------------- -----------------------------------

**5. GO/NO-GO判定（所要: 30分）**

**5-1. 集計**

5ページ × 3提案 =
15提案（レビュアーが1名の場合）。全提案の回答を集計する。

  --------------- ------------------------------------------- ---------------------------
  **判定**        **条件**                                    **次のアクション**

  GO ✅           「やりたい」が全提案の60%以上（9/15以上）   F0開発開始（Week 1へ）

  条件付きGO ⚡   「やりたい」が40-60%（6-9/15）              プロンプト修正 →
                                                              再テスト1回

  NO-GO ❌        「やりたい」が40%未満（6/15未満）           入力データ拡張 or
                                                              根本的再設計
  --------------- ------------------------------------------- ---------------------------

**5-2. GOの場合**

+-----------------------------------------------------------------------+
| **STEP GO-1: F0開発に進む**                                           |
|                                                                       |
| 1\. テスト結果をチーム全員に共有                                      |
|                                                                       |
| 2\. レビュアーの「微妙」コメントを集約 → プロンプト微調整リスト化     |
|                                                                       |
| 3\. Week 1のタスクを開始（拡張の骨格 + APIプロキシ + Side Panel UI）  |
|                                                                       |
| 4\. 開発と並行してプロンプト微調整を継続                              |
+-----------------------------------------------------------------------+

**5-3. 条件付きGOの場合**

+-----------------------------------------------------------------------+
| **STEP CONDITIONAL-1: プロンプトを修正して再テスト（+半日）**         |
|                                                                       |
| 1\. 「微妙」「やらない」の理由を分類:                                 |
|                                                                       |
| \- 「抽象的すぎる」→ プロンプトに具体性の指示を追加                   |
|                                                                       |
| \- 「的外れ」→ FVテキストの読み取り精度を確認                         |
|                                                                       |
| \- 「当たり前すぎる」→ プロンプトに「一般論は避ける」指示を追加       |
|                                                                       |
| 2\. proxy-worker.js の buildAnalyzePrompt() を修正                    |
|                                                                       |
| 3\. 同じ5ページで再テスト                                             |
|                                                                       |
| 4\. 再テストでも40%未満ならNO-GOへ                                    |
+-----------------------------------------------------------------------+

**5-4. NO-GOの場合**

+-----------------------------------------------------------------------+
| **STEP NOGO-1: 根本原因を特定**                                       |
|                                                                       |
| 1\. 「やらない」の理由を全件確認                                      |
|                                                                       |
| 2\. 以下のどれに該当するか判断:                                       |
|                                                                       |
| パターンA: 「ページの内容を読み取れていない」                         |
|                                                                       |
| → content-script.js                                                   |
| のFVテキスト抽出を改善。要素数・テキスト長の上限を引き上げ。          |
|                                                                       |
| パターンB: 「読み取れているが提案が浅い」                             |
|                                                                       |
| →                                                                     |
| プロンプ                                                              |
| トの分析指示を具体化。「なぜそれが問題なのか」の推論ステップを追加。  |
|                                                                       |
| パターンC: 「提案の方向は正しいがBefore/Afterが使えない」             |
|                                                                       |
| →                                                                     |
| プロ                                                                  |
| ンプトの出力フォーマット指示を改善。具体的なテキスト例の生成を強化。  |
|                                                                       |
| 3\.                                                                   |
| 修正後、                                                              |
| 再テスト。それでもNO-GOなら、v3.0のアプローチ自体を見直す必要がある。 |
