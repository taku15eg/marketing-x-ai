1\. 本質的な目的

**プロダクト目的:**
ページ改善の「判断」が、属人的な勘でも外部への丸投げでもなく、根拠と構造に支えられて前に進み、その結果が自動的に学習資産として積み上がる------このサイクルが、マーケター1人でも回り続ける状態をつくること。

**事業目的:**
14案件のコンサルティングで繰り返してきた「意思決定の詰まりを解消して前に進める」という価値を、プロダクトとして再現可能にし、コンサルの手が届かない数千〜数万のマーケターにスケールさせること。

1.1 避けたい状態

「便利な分析ツール」止まり：分析は出るが判断まで至らず「で、どうすればいいの？」が残る。

「提案の質が低くて使われなくなる」：ビジネス文脈を推定できず、的外れな提案が出る。

「ログは溜まるが学習が回らない」：記録はされるが次の判断に活かされず、ただのアーカイブになる。

Layer 0 → 1 転換のゲート設計（v3.0の核心）

Layer
0では分析結果・提案・Before/Afterはすべて閲覧可能。依頼パックも「プレビュー」として内容が見える。ただしコピーボタンが無効で「アカウント作成でコピー可能」と表示。行動を遮断するのではなく、行動の最後の1ステップだけをゲートにする。

5\. AI入力データの拡張（v3.0最重要変更）

**v2.0の最大の構造矛盾:**
AIがページの本文を読んでいなかった。これは「本を読まずに目次だけ見て書評を書く」のと同じ。

v3.0で追加する入力データ

1\. FVテキスト（上位1000pxのテキスト要素、最大30要素×200文字）

2\. CTA周辺コピー（各CTAの親セクションのテキスト、最大400文字）

3\. 見出し直下のテキスト（各h1-h3の次の段落、最大200文字）

トークン増加量は約500-1000トークン。API原価への影響は¥2-5/リクエスト。Layer
0の品質が劇的に上がることを考えれば許容範囲。

Content Script追加抽出コード（概要）

extractFVText(): 上位1000px内の要素からテキストを抽出。mask()でPII除去。

extractCTAContext(ctaElement): CTAの親セクションのテキストを取得。

extractHeadingContext(): h1-h3直下の1段落を取得。

6\. プロンプト設計（v3.0変更点）

フレーミング変更

v2.0: 「Webページの公開判断を支援するエキスパート」

v3.0: 「Webページの改善を支援するマーケティングエキスパート」

出力形式変更

v2.0: judgment = PASS/FAIL/HOLD（合否判定）

v3.0:
judgmentフィールドを廃止。すべて「改善提案」として出力。禁止ゾーン該当時のみHOLD判定を残す。

根拠チェーン（v3.0新設）

各提案に4ステップの根拠を必須化: ①現状（ページから読み取った事実）→
②業界傾向（AI知識ベース）→ ③ギャップ（①と②の差分）→
④仮説（改善の根拠）。各ステップにデータソースタグ（FVテキスト/CTA周辺/ページ構造/AI知識ベース/推定）を付与。

禁止事項（v3.0追加）

「もっと目立たせましょう」程度の汎用提案は出さない。ターゲットのニーズに言及しない提案は出さない。ページに書いてある事実に基づかない提案は出さない。

7\. 実験ノート型ログ（v3.0新設）

**v2.0のログ構造の問題:**
「PASS/理由/結果」しかなく、何をどう変えたか（Before/After）が入っていない。AIは因果関係を学習できない。

v3.0のログ構造

自動記録される情報: page_url, proposal_title, category, before, after,
target_selector, recorded_at

ユーザーが入力する情報:
実行状況（実行した/保留/見送り）、効果評価（5段階）、数値変化（CVR/CTR等、任意）

10件溜まるとAIが「この会社ではtrustカテゴリの改善が効きやすい。特にFVに数値を入れるパターンが2回成功している」と推論可能。これがMOATになる。

提案カードの構成

メタ情報: カテゴリタグ（信頼性/CTA/構成）+ 影響度タグ（大/中/小）

タイトル: ページの実テキストを引用した具体的な変更内容

展開時: ①ページの文脈 → ②根拠チェーン4ステップ → ③Before/After →
④期待効果 → ⑤注意点 → ⑥依頼パック（デザイナー/エンジニア切替）

依頼パックの構成

デザイナー向け: 変更の目的 → 変更内容（Before/After表） → チェックリスト

エンジニア向け: 変更箇所（セレクタ + Before/After表） → テスト要件

CTAの優先度変更（v3.0の核心）

v2.0: \[即時反映\] primary \[依頼パック\] secondary \[保留\] text

v3.0: \[プレビュー\] secondary \[依頼パックを作る\] primary
\[メモを残す\] text

「即時反映」はプレビュー機能として残すが、主役を「依頼パック」にする。依頼パックこそが第3層ペイン（分業の摩擦）の直接的な解消。

12\. Phase 0 検証テスト（開発着手前・必須）

**このテストをスキップして3週間開発するのと、このテストをやってから3週間開発するのでは、3週間後に手元にあるものの価値がまるで違う。**

手順（1日で完了）

1\. 手動で5サイトのDOM + FVテキスト + CTA周辺コピーを収集（2時間）

2\. v3.0プロンプトでClaude APIに投げる（1時間）

3\.
出力を藤野型ペルソナ相当の人（代理業/フリーランス）に見せて「この提案、クライアントにそのまま見せられる品質か？」と聞く（1時間）

4\.
3/5が「見せられる」と言えば開発GO。言わなければプロンプトを修正してもう1回。
