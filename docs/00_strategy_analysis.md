## 1. 結論：最上位の目的

### プロダクトの目的

**「何を聞くべきかすら知らない人」のために、「聞かなくても答えが出る」プロダクト。**

1つのURLを入れるだけで、そのビジネスのマーケティング全体像——LP改善・広告訴求・市場傾向・流入構造・競合・事業分析——が見え、次にやるべきことが全て出る。

### 事業の目的

プロダクト単体で勝つ。SEVENDEXのコンサル知見を内包しつつ、SEVENDEXに依存しない独立したSaaSとして、ブートストラップ・利益重視（Ahrefsモデル）で成長する。

### v2.0からの転換

| | v2.0（2/28） | v3.0（3/8） |
|--|--|--|
| 目的の主語 | 「判断を前に進める」 | 「判断そのものを不要にする」 |
| ターゲット | マーケを知っている人を助ける | **マーケを知らない人の代わりに判断する** |
| MOATの主語 | SoR（学習資産の蓄積） | **結果の深さ + 構造的模倣防御** |
| 本体 | Chrome拡張 | **Webダッシュボード** |
| スコープ | LP分析のみ | **1つのURLからマーケティング全体像（6タブ）** |

## 2. 根拠：6回の方向転換

Phase 0検証（3/7-8）と戦略ディスカッション（3/8）で6回の根本的転換が起きた。全て不可逆。

| # | 転換 | 表面の変更 | 裏にある本質 |
|---|------|-----------|-------------|
| 1 | DOMのみ→Vision API必須 | 技術要件追加 | プロダクトはユーザーの代わりに「見る」必要がある |
| 2 | コピー生成→構造変化図示 | 出力変更 | 「答え」ではなく「判断の材料」を出す |
| 3 | SoR主語→SoR副産物 | MOAT変更 | ユーザーは記録したいのではなく次にやることを知りたい |
| 4 | バイブマーケ向け→その先 | ターゲット変更 | 「問いを立てる力」の代替がプロダクトの価値 |
| 5 | Chrome拡張本体→Webダッシュボード本体 | 形態変更 | 成長は「成果物が外に出る」構造で決まる |
| 6 | LP分析のみ→6タブ拡張 | スコープ拡大 | 1つのURLからビジネスのマーケ全体像が見える |

### Phase 0検証で確定した事実

- **Vision API必須**: 日本LPは画像ベースが主流。DOMだけでは分析品質が出ない
- **4ステップパイプライン**: 企業リサーチ→Vision+DOM→診断→依頼パック。この順序で品質が確保される
- **薬機法チェック必須**: 健康食品/化粧品LPで届出とLP訴求のズレを検知できた
- **検証LP**: SEVENDEX LP、喜界島薬草農園LP

## 3. 摩擦の4層（不変）

3名のインタビュー原文から抽出された構造。立場もスキルレベルも違う3名が全員同じ4層を語っている。

| 層 | 摩擦 | 実声 | PGが解くメカニズム |
|---|------|------|-------------------|
| 第1層 | 要因の不可視 | 「なんでその数字になっているのかがわからない」（浦田型） | URL入力→AI分析が要因を構造化して一覧表示 |
| 第2層 | 施策の不在 | 「確からしい課題仮説とかイシューの特定ができてない」（藤野型） | 課題→改善方向性→依頼書→広告訴求文まで自動生成 |
| 第3層 | 分業の摩擦 | 「マーケのこと詳しくない開発者に細かく指示書を書くのが面倒」（七條型） | デザイナー/エンジニア向け構造化依頼書を自動出力 |
| 第4層 | 学習の断絶 | 「施策を回してる時間より、ログを残す・報告書を作る時間の方が多い」（浦田型） | 使うだけで自然にログが蓄積（**SoRは副産物**） |

**v3.0での変更点**: 第4層の位置付け。旧設計ではSoR（学習の蓄積）がMOATの主語だった。v3.0では副産物。価値はあるが課金の主動機ではない。課金の主動機は結果の「深さ」。

## 5. ゴール状態（v3.0）

**需要側（浦田型マーケ担当）の「買いたい状態」**：
URLを入れるだけで「何が悪い／なぜ／どう直す／広告をどう出す／市場はどうなっているか」が全部出て、自分で考えなくても次にやるべきことがわかる状態。

**発注側（七條型マーケ責任者）の「買いたい状態」**：
チーム全員がPublish Gateをデフォルトツールとして使い、担当者の属人的な勘やコンサルへの依存なしに、改善判断の品質と速度が標準化されている状態。

**成長側（プロダクト自体）の「買いたい状態」**：
分析結果が共有URLとして外に出続け、受け取った人が「自分もやりたい」と新規登録し、そのユーザーがまた共有する——フライホイールが自走している状態。

## 6. 競争優位（MOAT）

### v3.0のMOAT構造

v2.0ではSoR（学習資産の蓄積）がMOATの主語だった。v3.0では**構造的に真似しにくい設計**がMOAT。

| MOAT | 内容 | なぜ真似しにくいか |
|------|------|-------------------|
| 日本LP構造パターンDB | 匿名化・集約された日本LP特有の構造パターン | 後発は同じデータ量を持てない。時間が必要 |
| Vision API + 薬機法チェック品質 | Phase 0で検証済みの分析パイプライン | プロンプトだけでは再現不可。同等の検証が必要 |
| 依頼書フォーマットの業界定着 | デザイナー/エンジニアが「この形式で」と言い出す | ネットワーク効果。乗り換えコストが発生 |
| 共有URLの外部残存 | SlackやDocsに分析結果URLが残る | 解約してもURLは残存。再検討時にPGに戻る |
| 一気通貫の範囲 | LP分析→広告→市場→流入→競合→事業 | 各領域の個別ツールはあるが統合は困難 |

### 競合との差分

| 競合 | PGとの差分 |
|------|-----------|
| dejam（直接競合、月10万+） | PGは一気通貫（分析→広告→市場→流入→競合→事業）。dejamはLP分析のみ |
| Claude/GPTに直接聞く | PGは「聞き方を知らなくてもURLだけで出る」。直接聞くには「問い」が必要 |
| 30characters等（広告コピー生成） | PGはLP分析と統合されている。単体の広告コピー生成ではない |
| Ahrefs（SEOダッシュボード） | PGはSEOではなくLP改善/Webマーケ全体 |
| 「何もしない」 | 最大の競合。PGは「聞かなくても出る」ことで行動閾値を極限まで下げる |

## 7. 避けたい状態

1. **「便利な分析ツール」止まり**: 分析は出るが判断まで至らず「で、どうすればいいの？」が残る
2. **「提案の質が低くて使われなくなる」**: ビジネス文脈を推定できず的外れな提案が出る
3. **「ログは溜まるが学習が回らない」**: 記録はされるが次の判断に活かされない
4. **「情報過多で浦田型が圧倒される」**: 6タブが同時に見えて固まる → 段階的開放で対処
5. **「分析ツール」のカテゴリに固定される**: dejam/VWOと同じ棚に入れられる → Phase移行でカテゴリを変える
6. **「薄いAIラッパー」と認識される**: Claude/GPTに聞けば同じと思われる → 4ステップパイプライン+パターンDBで差別化
