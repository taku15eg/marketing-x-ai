## v3.0 変更サマリー

- β対象を藤野型3-5名（代理業/フリーランス）に変更
- センターピン：「藤野型が分析結果URLをクライアントに送ったか」
- 成功指標：β5名中3名が共有URL生成1回以上、共有URL経由新規1件以上

## 1. 検証の核心

### センターピン：共有URL送付

**「藤野型が分析結果URLをクライアントに送ったか」**

これ1つが達成されれば、以下が同時に証明される：

1. 分析結果に「クライアントに見せる価値」がある
2. 共有URLの生成・送付体験が十分スムーズ
3. バイラルループの起点（Loop 1）が機能する
4. 藤野型の「報告時間 > 実務時間」の痛みが軽減される

12ドキュメントの計画より、1人の「これをクライアントに送った」のほうが確実な証拠。

## 2. 3つの仮説

| # | 仮説 | 検証方法 | 成功の定義 |
|---|---|---|---|
| H1 | URL入力だけで「おっ」と思う分析結果が出る | URL入力→分析完了→タブ1閲覧30秒以上 | 分析完了率 > 85%、タブ1閲覧率 > 70%（30秒以上） |
| H2 | 分析結果の共有URLがクライアントに送られる | 共有URL生成→実際に送付（ヒアリング確認） | β5名中3名が1回以上共有URL生成 |
| H3 | 共有URLを見たクライアントが自ら分析を開始する | 共有URL閲覧→新規URL入力 | 共有URL経由新規が1件以上発生 |

### 仮説間の依存関係

```
H1（おっ） → H2（送る） → H3（新規が来る）
    ↓              ↓              ↓
  品質証明    バイラル起点    ループ成立
```

H1が成り立たなければH2は検証不能。H2が成り立たなければH3は検証不能。
Week 1-2でH1を集中検証し、Week 3-4でH2-H3を検証する。

## 3. β対象：藤野型3-5名

### 藤野型の定義

- 代理業またはフリーランスのマーケター/コンサルタント
- 複数クライアントのLP改善・広告運用を担当
- 「報告時間 > 実務時間」の痛みを持つ
- クライアントに分析結果・改善提案を共有する業務がある

### 募集要件

| 項目 | 要件 |
|---|---|
| 担当クライアント数 | 2社以上 |
| LP改善の実務経験 | あり |
| 分析ツール利用経験 | Ahrefs/SimilarWeb/GA等いずれか |
| β期間中の利用コミット | 週2回以上のURL入力、週1回のヒアリング対応 |

### 募集チャネル

- 既知の代理業/フリーランスへの直接声かけ（3名）
- X/LinkedIn経由（2名、バッファ）

### 不適格（除外対象）

- 社内マーケ専任（クライアントへの共有がない）
- 分析ツールを使ったことがない（比較対象がない）
- 週1回以上のヒアリングに対応できない

## 4. Week 1-4 計画

### Week 1：H1集中検証「おっ」の確認

**目的**：URL入力→分析結果で「おっ」が出るか

**やること**：
- β参加者3-5名にWebダッシュボードのURLを送付
- 各自の担当クライアントLP 3-5本をURL入力してもらう
- 初回のURL入力はZoom同席で観察（画面共有）
- 2回目以降は非同期で利用

**計測**：
- URL入力完了率
- 分析パイプライン成功率
- タブ1（LP分析）閲覧時間
- 初回「おっ」の瞬間（Zoom観察で記録）
- 定性フィードバック（毎日5分Slackヒアリング）

**Week 1末チェック**：
- 分析完了率 > 85% → 続行
- 分析完了率 < 70% → パイプライン修正に1週間投入、Week 2をリセット
- 「おっ」が0名 → プロンプト・Vision API分析の根本改善

### Week 2：H1深掘り＋H2準備

**目的**：分析品質の安定化、共有URL生成の自然発生を観察

**やること**：
- 各参加者が新規LP 5-10本を追加分析
- Week 1のフィードバックを反映した改善版をデプロイ
- 共有URL機能の案内（「クライアントに見せたいと思ったら共有ボタンを押してください」）
- 依頼書ダウンロード体験の確認

**計測**：
- 2回目以降のURL入力頻度（自発的利用率）
- 分析結果のどの部分を長く見ているか（タブ内スクロール深度）
- 共有URL生成の有無（まだ強制しない）
- 「これクライアントに見せたい」の発言（ヒアリング）

**Week 2末チェック**：
- 自発的利用が週2回以上 → H1検証成功、H2検証へ進む
- 自発的利用が週1回未満 → UXの障壁特定、改善して延長

### Week 3：H2検証「送る」の確認

**目的**：共有URLが実際にクライアントに送られるか

**やること**：
- 「次のクライアント報告で使ってみてください」と依頼
- 共有URL生成→送付のフローをサポート
- 送付先クライアントの反応をヒアリング

**計測**：
- 共有URL生成数
- 共有URL送付数（ヒアリングで確認、計測値との差異も確認）
- 送付方法（メール、Slack、LINE、口頭URLコピー等）
- クライアントの反応（定性）

**Week 3末チェック**：
- 3名以上が共有URL生成 → H2検証成功、H3検証へ進む
- 1-2名のみ → 生成しなかった理由を特定、改善
- 0名 → 共有の心理的障壁を分析、根本的なアプローチ変更を検討

### Week 4：H3検証「新規が来る」の確認＋GO/NO-GO判定

**目的**：バイラルループの成立確認

**やること**：
- Week 3で送付された共有URLの閲覧状況を追跡
- 共有URL経由で新規分析が発生したか確認
- β全体の振り返りヒアリング（各30分）
- GO/NO-GO判定会議

**計測**：
- 共有URL閲覧数
- 共有URL経由の新規URL入力数
- β参加者のNPS（Net Promoter Score）
- 「これなしでは仕事したくない」の発言数

## 5. GO/NO-GO判定（Week 4末）

### GO条件（Phase 1に進む）

**必須条件（すべて満たす）**：

| # | 条件 | 判定基準 |
|---|---|---|
| 1 | センターピン達成 | β5名中3名以上が共有URLを1回以上生成し、実際にクライアントに送付 |
| 2 | 分析品質 | 分析完了率 > 85%、タブ1閲覧率 > 70%（30秒以上） |
| 3 | バイラル起点 | 共有URL経由の新規分析が1件以上発生 |

**強いGO信号（1つでもあれば自信を持ってGO）**：
- 「いくら払えばもっと使える？」と聞かれる
- クライアント側から「これ何？うちもやりたい」と来る
- 参加者が依頼されていないLPも自発的に分析している

### NO-GO条件（いずれか1つでも該当）

| # | 条件 | 次の一手 |
|---|---|---|
| 1 | 分析完了率 < 70%が2週間改善しない | パイプライン再設計（Vision API精度・DOM解析改善） |
| 2 | 「おっ」が2名以下 | プロンプト設計の根本見直し、分析の切り口変更 |
| 3 | 共有URL生成が1名以下 | 共有の心理的障壁を分析、出力フォーマット変更 |
| 4 | 「自分でやった方が早い」が過半数 | 対象ユーザー（藤野型）の再定義、ペインの再検証 |
| 5 | プロンプト漏洩・セキュリティ事故 | 即時停止、原因究明、セキュリティ設計の全面見直し |

### PIVOT条件

NO-GOでも以下が観察されたら方向転換を検討：
- 藤野型は使わないが、浦田型（事業会社マーケ）には刺さっている → ターゲット変更
- 共有はしないが、個人利用で「手放せない」 → 共有よりも深度をNorth Starに変更
- LP分析は弱いが、広告訴求（タブ2）に強い反応 → Phase 1のタブ2を前倒し

## 6. β運営体制

| 役割 | 担当 | 稼働 | 責任 |
|---|---|---|---|
| β PM / PO | プロダクトオーナー | 週10h | GO/NO-GO判定、優先度決定 |
| エンジニア | 開発 | 週30h | パイプライン修正、バグ修正、計測実装 |
| βサポート | ユーザー対応 | 週8h | ヒアリング、Slack対応、定性データ収集 |

### ヒアリング設計

| 頻度 | 方法 | 内容 | 所要時間 |
|---|---|---|---|
| 初回（Week 1） | Zoom同席 | 初回URL入力の観察、「おっ」の瞬間記録 | 30分 |
| 日次（Week 1-2） | Slack非同期 | 「今日使いましたか？」「何がよかった/微妙？」 | 5分 |
| 週次（Week 2-4） | Zoom | 利用状況振り返り、共有URL送付の有無確認 | 15分 |
| 最終（Week 4） | Zoom | 全体振り返り、NPS、継続利用意向、課金意向 | 30分 |

### コミュニケーションツール

- **Slack**：β参加者専用チャンネル（#publish-gate-beta）
- **Zoom**：ヒアリング
- **Notion**：定性フィードバック蓄積
- **Webダッシュボード内計測**：定量データ（Supabase + イベント計測）

### エスカレーションフロー

```
バグ報告（Slack）→ エンジニア確認（2h以内）→ 修正デプロイ（24h以内）
品質問題（ヒアリング）→ PO判断（翌営業日）→ プロンプト改善（48h以内）
セキュリティ事故 → 即時停止 → PO + エンジニア緊急対応（30分以内）
```
