# Publish Gate β検証計画書

**版：v4.0　更新日：2026-03-15（戦略再設計に伴う全面改訂）**

---

## v4.0 変更サマリー

- **センターピンを変更**: 「共有URL送付」→「診断品質が行動を変えるか」
- **Phase 0.5の役割を再定義**: 製品リリースではなく「1仮説の検証装置」
- **Chrome拡張をPhase 1に移動**: 5名βでの仮説検証にはWebのみで十分
- **Day 0テストを新設**: β前に実LP5本で品質基準を確認
- **KPI優先順位を変更**: 診断品質 > 行動変容 > リテンション > 共有

## 1. 検証の核心

### 仮説（1つだけ）

> **「AIが生成したLP診断結果が、プロのマーケター（藤野型）の次のアクションを実際に変えるか」**

具体的には:
1. 分析結果を見て「これは正しい」と感じるか（品質）
2. 分析結果を見て「これを使おう」と思うか（行動変容）
3. 分析結果を他者に見せたいと思うか（共有意欲）

### なぜ「共有URL送付」ではなくなったか

共有URL送付はバイラル指標として重要だが、品質が伴わなければ共有は起きない。
センターピンは**原因**（品質）であるべきで、**結果**（共有）ではない。
品質が高ければ共有は自然に起きる（藤野型はクライアントに送りたいから）。

## 2. 4つの仮説（優先順）

| # | 仮説 | 検証方法 | 成功の定義 |
|---|---|---|---|
| H0 | 分析結果の品質が「実際に使える」レベルに達している | Day 0テスト（β前。実LP5本で品質検証） | 5本中4本で「アクション可能な指摘」が含まれる |
| H1 | 藤野型が分析結果を「自分でやるより速い・同等以上」と評価する | β参加者ヒアリング + 2回目分析の有無 | β5名中3名が「実際に使える」と評価 |
| H2 | 藤野型が分析結果に基づいて行動を変える（依頼書コピー/改善着手/チーム共有） | イベントログ + ヒアリング | β5名中3名が何らかのアクションを取る |
| H3 | 藤野型が分析結果を他者（クライアント/チーム）に共有する | 共有URL生成ログ + ヒアリング | β5名中2名が共有URLを生成 |

### 仮説間の依存関係

```
H0（品質基準） → H1（品質評価） → H2（行動変容） → H3（共有）
    ↓                  ↓                ↓               ↓
 βに進めるか      プロダクト価値    業務組み込み    バイラル起点
```

H0はβ前のゲート。H0をクリアしなければβは開始しない。

## 3. Day 0テスト（β前の品質検証）

### 目的

β参加者の貴重な時間を使う前に、分析品質が最低基準に達しているか確認する。

### 方法

1. 実際のLP 5本を選定（BtoB SaaS 2本、EC/D2C 1本、コンサル 1本、健康食品 1本）
2. Publish Gateで分析を実行
3. 分析結果を以下の基準で評価:

| 基準 | 合格条件 |
|------|---------|
| 企業理解の正確性 | 業界・事業モデルが概ね正しい |
| ページ構造の読解 | FV・CTA・信頼要素が正しく検出されている |
| 課題の的確性 | 指摘された課題が「プロのマーケターが見ても納得する」レベル |
| 依頼書の行動可能性 | 依頼書を読んだデザイナー/エンジニアが「何をすべきか理解できる」 |
| 薬機法検知（該当LPのみ） | リスク表現を検知できている |

### 合格条件

- 5本中4本以上で「課題の的確性」「依頼書の行動可能性」が合格
- 合格しない場合: プロンプト改善に1-2週間投入し、再テスト

### 不合格時の対応

| 問題 | 対応 |
|------|------|
| 企業理解が不正確 | company-research.tsのfetch先・プロンプト改善 |
| ページ構造の読解ミス | Vision APIの解像度・DOM抽出ロジック改善 |
| 課題が的外れ | 診断プロンプトの業界コンテキスト強化 |
| 依頼書が抽象的 | handoffプロンプトの具体性向上 |

## 4. β対象：藤野型3-5名

### 藤野型の定義

- 代理業またはフリーランスのマーケター/コンサルタント
- 複数クライアントのLP改善・広告運用を担当
- 「報告時間 > 実務時間」の痛みを持つ
- クライアントに分析結果・改善提案を共有する業務がある

### 募集要件

| 項目 | 要件 |
|---|---|
| 担当クライアント数 | 2社以上 |
| LP改善の実務経験 | あり |
| 月のLP分析頻度 | 月2回以上 |
| 分析ツール利用経験 | Ahrefs/SimilarWeb/GA等いずれか |
| β期間中の利用コミット | 週2回以上のURL入力、週1回のヒアリング対応 |

### 募集チャネル

- 既知の代理業/フリーランスへの直接声かけ（3名）
- X/LinkedIn経由（2名、バッファ）

### 不適格（除外対象）

- 社内マーケ専任（クライアントへの共有がない）
- 分析ツールを使ったことがない（比較対象がない）
- 週1回以上のヒアリングに対応できない

## 5. β期間：2週間（Day 0テスト後）

### Week 1：H1検証「品質は十分か」

**目的**：分析結果が「実際に使える」と評価されるか

**やること**：
- β参加者3-5名にWebダッシュボードのURLを送付
- 各自の担当クライアントLP 3-5本をURL入力してもらう
- 初回のURL入力はZoom同席で観察（画面共有）
- 2回目以降は非同期で利用

**計測**：
- URL入力完了率
- 分析パイプライン成功率
- 課題カード展開率（どの課題を深掘りしたか）
- 依頼書パネルの表示率
- 初回「おっ」の瞬間（Zoom観察で記録）
- 定性フィードバック（毎日5分Slackヒアリング）

**Week 1末チェック**：
- 3名以上が「実際に使える」と評価 → H1検証成功、H2検証へ
- 1-2名のみ → 品質問題を特定、プロンプト改善して延長
- 0名 → プロンプト設計の根本見直し、Day 0テストに戻る

### Week 2：H2-H3検証「行動は変わるか・共有するか」

**目的**：分析結果が行動変容と共有を引き起こすか

**やること**：
- 各参加者が新規LP 5-10本を追加分析（自発的利用を観察）
- 依頼書の活用状況を確認（「デザイナーに送りましたか？」）
- 共有URL機能の案内（「クライアントに見せたいと思ったら共有ボタンを」）
- 最終ヒアリング（各30分）

**計測**：
- 2回目以降のURL入力頻度（自発的利用率）
- 依頼書コピー/ダウンロード数
- 共有URL生成数
- NPS（Net Promoter Score）
- 「これなしでは仕事したくない」の発言
- 課金意向（「いくらなら払いますか？」）

## 6. GO/NO-GO判定（β終了時）

### GO条件（Phase 1に進む）

**必須条件（すべて満たす）**：

| # | 条件 | 判定基準 |
|---|---|---|
| 1 | 診断品質 | β5名中3名以上が「実際に使える」と評価 |
| 2 | 行動変容 | β5名中3名以上が何らかのアクション（依頼書コピー/改善着手/共有）を実行 |
| 3 | リテンション | β5名中3名以上が2回目の分析を自発的に実施 |

**強いGO信号（1つでもあれば自信を持ってGO）**：
- 「いくら払えばもっと使える？」と聞かれる
- クライアント側から「これ何？うちもやりたい」と来る
- 参加者が依頼されていないLPも自発的に分析している

### NO-GO条件（いずれか1つでも該当）

| # | 条件 | 次の一手 |
|---|---|---|
| 1 | 「実際に使える」評価が2名以下 | プロンプト設計の根本見直し、分析の切り口変更 |
| 2 | 行動変容が1名以下 | 出力フォーマットの変更（依頼書の粒度・具体性向上） |
| 3 | 「自分でやった方が早い」が過半数 | 対象ユーザー（藤野型）の再定義、ペインの再検証 |
| 4 | 「ChatGPTに聞くのと同じ」が過半数 | 依頼書レイヤーの強化、業界特化度の向上 |
| 5 | プロンプト漏洩・セキュリティ事故 | 即時停止、原因究明、セキュリティ設計の全面見直し |

### PIVOT条件

NO-GOでも以下が観察されたら方向転換を検討：
- 藤野型は使わないが、浦田型（事業会社マーケ）には刺さっている → ターゲット変更
- 共有はしないが、個人利用で「手放せない」 → 共有よりも深度をNorth Starに変更
- LP分析は弱いが、依頼書生成に強い反応 → 「AI依頼書ジェネレーター」にピボット
- LP分析は弱いが、広告訴求（タブ2）に強い反応 → Phase 1のタブ2を前倒し

## 7. β運営体制

| 役割 | 担当 | 稼働 | 責任 |
|---|---|---|---|
| β PM / PO | プロダクトオーナー | 週10h | GO/NO-GO判定、優先度決定 |
| エンジニア | 開発 | 週30h | パイプライン修正、バグ修正、計測実装 |
| βサポート | ユーザー対応 | 週8h | ヒアリング、Slack対応、定性データ収集 |

### ヒアリング設計

| 頻度 | 方法 | 内容 | 所要時間 |
|---|---|---|---|
| 初回（Week 1 Day 1） | Zoom同席 | 初回URL入力の観察、「おっ」の瞬間記録 | 30分 |
| 日次（Week 1） | Slack非同期 | 「今日使いましたか？」「何がよかった/微妙？」 | 5分 |
| 週次（Week 2） | Zoom | 利用状況振り返り、行動変容の確認 | 15分 |
| 最終（Week 2末） | Zoom | 全体振り返り、NPS、継続利用意向、課金意向 | 30分 |

### フィードバック収集の構造化

最終ヒアリングで必ず聞くこと:
1. 「分析結果で最も役に立った指摘は？」
2. 「分析結果で的外れだった指摘は？」
3. 「この結果をクライアント/チームに見せましたか？見せなかった理由は？」
4. 「ChatGPT/Claudeに直接聞くのと比べてどうですか？」
5. 「月額いくらなら使い続けますか？」
6. 「これがないと仕事で困りますか？」

### コミュニケーションツール

- **Slack**：β参加者専用チャンネル（#publish-gate-beta）
- **Zoom**：ヒアリング
- **Notion**：定性フィードバック蓄積
- **Webダッシュボード内計測**：定量データ（event-logger.ts）

### エスカレーションフロー

```
バグ報告（Slack）→ エンジニア確認（2h以内）→ 修正デプロイ（24h以内）
品質問題（ヒアリング）→ PO判断（翌営業日）→ プロンプト改善（48h以内）
セキュリティ事故 → 即時停止 → PO + エンジニア緊急対応（30分以内）
```
