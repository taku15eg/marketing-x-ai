# Publish Gate 要件定義書

**版：v3.0　更新日：2026-03-08（全面刷新）**

---

## 1. 本質的な目的

**「何を聞くべきかすら知らない人」のために、「聞かなくても答えが出る」プロダクト。**

1つのURLを入れるだけで、LP改善・広告訴求・市場傾向・流入構造・競合・事業分析の全体像が見え、次にやるべきことが全て出る。

### 1.1 事業としての目的

プロダクト単体で勝つ。ブートストラップ・利益重視（Ahrefsモデル）。SEVENDEXのコンサル知見を内包しつつ、SEVENDEX依存なしで独立成長するSaaS。

### 1.2 v2.0→v3.0の転換

| | v2.0 | v3.0 |
|--|------|------|
| 目的の主語 | 「判断を前に進める」 | **「判断そのものを不要にする」** |
| ターゲット | マーケを知っている人を助ける | **マーケを知らない人の代わりに判断する** |
| 本体 | Chrome拡張 | **Webダッシュボード** |
| スコープ | LP分析のみ（Layer 0-3） | **6タブ（LP→広告→市場→流入→競合→事業）** |
| 課金の主語 | SoR蓄積＋連携深度 | **結果の「深さ」** |
| MOAT | 判断ログ（SoR） | **構造的模倣防御** |

## 2. ユーザーの4層ペイン

3名のインタビューから抽出。立場を超えて共通する4層。

| 層 | 摩擦 | 代表的な発言 | PGが解くメカニズム |
|---|------|-------------|-------------------|
| 第1層 | 要因の不可視 | 「なんでその数字になっているのかがわからない」（浦田型） | URL入力→AI分析が要因を構造化して一覧表示 |
| 第2層 | 施策の不在 | 「確からしい課題仮説とかイシューの特定ができてない」（藤野型） | 課題→改善方向性→依頼書→広告訴求文まで自動生成 |
| 第3層 | 分業の摩擦 | 「マーケのこと詳しくない開発者に細かく指示書を書くのが面倒」（七條型） | デザイナー/エンジニア向け構造化依頼書を自動出力 |
| 第4層 | 学習の断絶 | 「施策を回してる時間より、ログを残す・報告書を作る時間の方が多い」（浦田型） | 使うだけで自然にログが蓄積（SoRは副産物） |

## 3. 3アーキタイプ

| Type | Pain | 最大価値 | 位置付け |
|------|------|---------|----------|

| 七條型（マーケ責任者） | 教育コスト | チーム標準化 | Business層 |

## 4. 設計原則

### 4.1 10の核心原則

1. **URL入力だけで全部出る。追加入力を要求しない**
2. **Webダッシュボードが本体。Chrome拡張はリテンション/リマーケ**
3. **Ahrefs Site Explorer型UI（URL入力→結果一覧→深掘り）。チャットUIではない**
4. **分析結果は共有可能なURLとして生成。バイラルの起点**
5. **全出力物にPublish Gate導線（依頼書Powered by等）**
6. **Vision API必須。日本LPは画像ベースが主流**
7. **薬機法/景品表示法チェック必須**
8. **コピー文言は出さず構造変化を図示する**
9. **SoRは副産物。課金の主動機は結果の「深さ」**
10. **Phase展開：0.5→1→2→3（センターピン駆動）**

### 4.2 UI思想

- Ahrefs Site Explorer型：URL入力バー→結果一覧→クリックで深掘り
- タブは存在するが、最初はタブ1だけがアクティブ。他タブはロックアイコン＋「Starterで解放」表示
- 各タブ間のストーリー接続：LP分析の課題から「広告でも使うなら？→タブ2」「市場は成長してるか？→タブ3」と自然誘導
- 浦田型が圧倒されない：情報を絞り、深掘りはアコーディオン展開
- 各課題に「次にやること」が必ず付く（分析ツール止まりを回避）

## 5. ゴール状態（v3.0）

**需要側（浦田型）**: URLを入れるだけで「何が悪い／なぜ／どう直す／広告をどう出す／市場はどうなっているか」が全部出て、自分で考えなくても次にやるべきことがわかる状態。

**発注側（七條型）**: チーム全員がPGをデフォルトツールとして使い、改善判断の品質と速度が標準化されている状態。

**成長側（プロダクト）**: 分析結果URLが外に出続け、受け取った人が新規登録→また共有→フライホイールが自走している状態。

## 6. ダッシュボード6タブ仕様

### 6.1 タブ構造

| Tab | 内容 | 解放Plan | データソース | Phase |
|-----|------|---------|-------------|-------|
| 1. LP分析 | 課題一覧→詳細→依頼書→計測設計→薬機法チェック | Free〜 | URL入力（Vision+DOM） | 0.5 |
| 2. 広告訴求 | Google/Meta/PMax向け訴求文、ターゲット推薦 | Starter〜 | LP分析結果から逆算 | 1 |
| 3. 市場分析 | 流入クエリ、検索ボリューム推移、市場傾向示唆 | Pro〜 | GSC＋キーワードプランナー | 2 |
| 4. 流入分析 | 参照元ランキング、セッション占有率、媒体別 | Pro〜 | GA4 | 2 |
| 5. 競合分析 | 競合LP構造比較、訴求差別化ポイント | Pro〜 | AI推定＋公開情報 | 2 |
| 6. 事業分析 | 市場規模推定、顧客単価、事業モデル推定 | Business〜 | AI推定＋業界データ | 3 |

### 6.2 タブ1（LP分析）詳細 — Phase 0.5 MVP

コア体験：URL入力→数秒→結果一覧

**分析エンジン（4ステップパイプライン）：**
1. 企業リサーチ — URL→企業情報・業界・事業モデルを推定
2. Vision + DOM分析 — スクリーンショット（Vision API）＋DOM構造の両面分析
3. 診断 — 業界コンテキスト×構造分析→課題抽出・優先順位付け
4. 依頼パック生成 — デザイナー/エンジニア向け構造化依頼書＋計測設計＋薬機法チェック

**タブ1の出力物：**
- 課題一覧（概要 = Free / 詳細 = Starter）
- 改善方向性（構造変化の図示。コピー文言は出さない）
- デザイナー向け依頼書（Starter〜）
- エンジニア向け依頼書（Starter〜）
- 計測設計（Pro〜）
- 薬機法/景品表示法チェック結果（Starter〜）

**タブ1で作らないもの：**
- ABテスト自動化
- ヒートマップ連携
- ページ上オーバーレイ
- 即時反映（Preview Apply）— v3.0ではWebダッシュボードが本体のため不要

### 6.3 タブ2（広告訴求）詳細 — Phase 1

LP分析結果から逆算して広告訴求文を生成。

**出力物：**
- Google Ads RSA（レスポンシブ検索広告）向け見出し＋説明文
- Google PMax向け訴求アセット
- Meta Ads向け訴求文（文字数制約準拠）
- ターゲット推薦（LP分析からの推定ペルソナ）
- CSVエクスポート（Google Ads Editor互換フォーマット）

### 6.4 タブ3-4（市場/流入）詳細 — Phase 2

**タブ3（市場分析）：**
- GSCデータ → 流入クエリ一覧 → 検索ボリューム推移
- キーワードプランナー連携（広告アカウント経由）
- 市場傾向の示唆（「このカテゴリの検索は〇〇%成長」等）

**タブ4（流入分析）：**
- GA4データ → 参照元ランキング → セッション占有率
- 媒体別のパフォーマンス比較
- 流入チャネルの偏りに対する示唆

**Phase 2初期はCSV手動アップロードで検証可能。OAuth直結はPhase 2中盤。**

### 6.5 タブ5（競合分析）詳細 — Phase 2

- 競合LP URL入力（最大3社 = Pro / 無制限 = Business）
- LP構造の比較（FV・CTA配置・訴求構造の差分）
- 訴求差別化ポイントの抽出
- 「競合がやっていてこのLPがやっていないこと」の列挙

### 6.6 タブ6（事業分析）詳細 — Phase 3

- 市場規模推定（AI推定＋業界データ）
- 想定顧客単価・LTV推定
- 事業モデルの推定と示唆
- 投資対効果のシミュレーション

## 7. Chrome拡張仕様

### 7.1 位置付け

**リテンション/リマーケ用。本体ではない。** Webダッシュボードの簡略版としてSide Panelで提供。同じAPIを叩く。

### 7.2 機能

- LP閲覧時にアイコン色変化（LP検知ロジック）
- ワンクリックで現在のページをWebダッシュボードで分析開始
- Side Panelで最新の分析結果サマリー表示
- バッジ通知（2週間に1回、定期チェック促進。月最大4回）
- 月次サマリー通知

### 7.3 Chrome拡張で作らないもの

- 完全な分析フロー（Webダッシュボードに誘導）
- アカウント管理
- 依頼書の詳細表示
- CSVエクスポート

## 8. 共有URL仕様

### 8.1 概要

分析結果を共有可能なURLとして生成。**バイラルの最強経路。**

### 8.2 仕様

- 分析完了時に自動的に共有URL生成
- URLにアクセスすると分析結果のRead-Only版が表示
- 閲覧にアカウント不要（Free相当の概要のみ表示）
- 「自分のLPも分析する」CTAを常時表示
- Publish Gate導線を明確に配置

### 8.3 制約

- 共有URLの有効期限：無期限（残存がMOATになる）
- Cloudflare Turnstile（CAPTCHA）設置（Phase 1〜）
- 月100回アクセス超過でアカウント要求（Phase 1〜）

## 13. 品質基準

### 13.1 分析品質

- **Vision API分析精度**: 主要要素（FV・CTA・フォーム）の検出率 > 90%（代表10サイト）
- **企業リサーチ精度**: 業界・事業モデル推定の正答率 > 80%
- **根拠なし提案 = 0件**: すべての課題に根拠と改善方向性が付く
- **薬機法チェック精度**: 健康食品/化粧品LPでの違反検知率 > 85%

### 13.2 安全基準（非交渉）

- 個人情報のマスキング漏れ：0件
- APIキーのクライアント露出：0件
- プロンプト漏洩：0件
- 共有URLからの個人情報露出：0件

## 14. セキュリティ要件

### 14.1 最重要原則

**Chrome拡張には「知性」を置かない。** DOM抽出とUI表示のみ。分析の核心（プロンプト・パイプライン・パターンDB）は全てサーバーサイドに閉じる。

### 14.2 Phase別セキュリティ要件

**Phase 0.5（必須）：**
- プロンプトインジェクション防御（URL入力バリデーション＋既知パターン検知）
- Claude APIへのプロンプト保護指示（システムプロンプト出力禁止）
- 出力の正規化（生レスポンスをそのまま返さない）

**Phase 1：**
- プロンプトのKV Store暗号化保存
- 多層レート制限（IP＋フィンガープリント＋user_id＋パターン検知）
- Cloudflare Turnstile（共有URLのスクレイピング防御）

**Phase 2：**
- Chrome拡張の難読化（obfuscator.io）
- Supabaseカラムレベル暗号化（pgcrypto）
- OAuthトークンのAES-256暗号化
- APIキーの月次自動ローテーション

## 15. 避けたい状態

1. **「便利な分析ツール」止まり** — 分析は出るが「で、どうすればいいの？」が残る
2. **「提案の質が低くて使われなくなる」** — ビジネス文脈を推定できず的外れ
3. **「ログは溜まるが学習が回らない」** — 記録はされるが次に活かされない
4. **「情報過多で浦田型が圧倒される」** — 6タブが同時に見えて固まる
5. **「分析ツール」のカテゴリに固定される** — dejam/VWOと同じ棚に入れられる
6. **「薄いAIラッパー」と認識される** — Claude/GPTに聞けば同じと思われる
