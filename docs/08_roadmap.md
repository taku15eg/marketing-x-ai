## 全体構成

```
Phase 0.5 β（W1-4）→ Phase 1 初期公開（W4-8）→ Phase 2 成長期（W8-16）→ Phase 3 拡張期（W16-24）
```

各Phaseの移行はセンターピン（1つの問い）の達成で判定する。時間ではなく行動で進む。

## Phase 0.5: β検証装置（Week 1-4）

### 概要

| 項目 | 内容 |
|------|------|
| 作るもの | Webダッシュボード（LP分析＋依頼書）＋共有URL＋Day 0テスト |
| 作らないもの | Chrome拡張、タブ2-6、アカウント機能、メール通知、CSVエクスポート |
| 対象 | 藤野型3-5名 |
| 役割 | **製品リリースではなく、1仮説の検証装置** |

> Chrome拡張はPhase 1に移動。5名βでの仮説検証にはWebのみで十分。

### 検証する仮説

**「AIが生成したLP診断結果が、プロのマーケター（藤野型）の次のアクションを実際に変えるか」**

### 成功指標（優先順）

1. 診断品質: β5名中3名が「実際に使える」と評価
2. 行動変容: β5名中3名が何らかのアクション実行
3. リテンション: β5名中3名が2回目分析を自発的に実施
4. 共有: β5名中2名が共有URL生成

### Week 1: Day 0テスト＋品質確認

| タスク | 担当 | 完了条件 |
|--------|------|---------|
| Day 0テスト用LP 5本選定 | PM | BtoB SaaS 2本、EC 1本、コンサル 1本、健康食品 1本 |
| 分析品質の手動評価 | PM/エンジニア | 5本中4本で「アクション可能な指摘」合格 |
| 品質不合格時のプロンプト改善 | エンジニア | 再テストで合格 |
| β配布準備（デプロイ+案内文作成） | PM | Vercelにデプロイ、URLが安定稼働 |

### Week 2: β開始＋H1検証

| タスク | 担当 | 完了条件 |
|--------|------|---------|
| β参加者3-5名にURL送付 | PM | 全員がアクセス可能 |
| 初回Zoom同席観察 | PM | 全員の初回URL入力を観察、「おっ」の瞬間記録 |
| 日次Slackヒアリング | PM | 品質フィードバック収集 |
| フィードバック反映の品質改善 | エンジニア | 主要指摘への対応デプロイ |

### Week 3: H2-H3検証＋追加分析

| タスク | 担当 | 完了条件 |
|--------|------|---------|
| 自発的利用の観察 | PM | 2回目以降のURL入力頻度を計測 |
| 依頼書活用状況の確認 | PM | 「デザイナーに送りましたか？」ヒアリング |
| 共有URL生成の自然発生を観察 | PM | 共有を強制せず、自然な共有意欲を確認 |
| 品質改善の追加デプロイ | エンジニア | Week 2のフィードバック反映 |

### Week 4: GO/NO-GO判定

| タスク | 担当 | 完了条件 |
|--------|------|---------|
| 最終ヒアリング（各30分） | PM | NPS、課金意向、改善要望の収集 |
| 定量データ集計 | エンジニア | イベントログの集計レポート |
| GO/NO-GO判定会議 | PM/PO | 判定結果とPhase 1計画の確定 |

### 移行条件（0.5→1）

- β5名中3名が診断品質を「実際に使える」と評価
- β5名中3名が何らかの行動変容（依頼書コピー/改善着手/共有）
- β5名中3名が2回目の分析を自発的に実施

## 技術デット管理

Phase 0.5-1では速度優先。以下はPhase 2以降に改善：

| 項目 | Phase 0.5-1 | Phase 2以降 |
|------|------------|------------|
| エラーハンドリング | 基本的なtry-catch | Sentry統合 |
| テスト | 手動テスト20ページ | E2Eテスト自動化（Playwright） |
| CI/CD | 手動デプロイ | GitHub Actions |
| パフォーマンス | 最適化なし | レスポンス時間の計測・改善 |
| ログ/モニタリング | console.log | 構造化ログ＋ダッシュボード |
| プロンプト管理 | proxy-worker内 | KV Store暗号化 |
