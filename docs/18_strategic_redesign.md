# Publish Gate 戦略再設計書 v4.0

**作成日**: 2026-03-15
**目的**: repo全体の戦略・プロダクト・KPI・実装の一貫性を再構築する

## 1. 結論

### 現状の最大問題

Publish Gateのrepoには**戦略的一貫性の欠如**がある。具体的には以下の3つの構造的問題が存在する。

1. **ターゲット定義の分裂**: CLAUDE.mdは「できたばかりのマーケ部門の人」（浦田型）を主ターゲットとするが、β計画は藤野型（代理業）を初手とし、Project Charterは「小規模広告代理店」と記載。3文書が3つの異なるターゲットを主語にしている。これは「初手市場」と「長期本命市場」の意図的分離が**明示的に定義されていない**ことが原因。

2. **Phase 0.5スコープの肥大化**: 「Webダッシュボード + 共有URL + Chrome拡張」をPhase 0.5に含めているが、5名βで検証すべき仮説は「分析結果が行動を変えるか」の1点であり、Chrome拡張は検証に不要。Phase 0.5の役割が「仮説検証」ではなく「ミニ製品リリース」になっている。

3. **センターピン指標の誤設定**: 成功条件が「3/5名が共有URL生成」だが、インタビューが示す最大のペインは「なぜその数字かわからない」「何をすればいいかわからない」であり、共有はペイン解消の結果であって目的ではない。共有URL生成率をセンターピンにすると、「共有しやすいUI」に最適化して「診断品質」がおろそかになるリスクがある。

### 新しい戦略の骨子

| 項目 | 旧（AS-IS） | 新（TO-BE） |
|------|------------|------------|
| 初手市場 | 藤野型（代理業） | 藤野型（代理業）**を明示的に「初手」と定義** |
| 長期本命 | 暗黙的に浦田型 | **浦田型（事業会社マーケ）を明示的に「本命」と定義** |
| Phase 0.5の役割 | ミニ製品リリース | **1仮説の検証装置** |
| センターピン | 共有URL生成 | **「分析後に行動変容が起きたか」** |
| Chrome拡張 | Phase 0.5必須 | **Phase 1に後退** |
| プロダクト定義 | LP分析ツール | **マーケ業務の診断→依頼の代行** |

### 何を捨てるか

- Phase 0.5からChrome拡張を除外（Phase 1で投入）
- Phase 0.5からソーシャルシェアボタンを除外
- Phase 0.5からタブ2-6のロックUI表示を除外（存在を示唆しない）
- Skill Orchestrator関連ファイル（SKILL.md, skill-orchestrator-architecture.md, EXECUTION-STEPS.md, CROSS-PLATFORM-GUIDE.md）をプロダクト文書体系から分離
- 「PASS/FAIL」ゲート判定の概念を完全撤廃（v2.0の残骸）

### 何を残すか

- 4ステップ分析エンジン（検証済み）
- 共有URL（ただしセンターピンではなくバイラル手段として）
- 藤野型β→浦田型本命の市場順序（ただし明示的に分離）
- 依頼書（Brief）レイヤー（差別化の核）
- セキュリティ要件（SSRF, プロンプトインジェクション防御）
- 価格4層構造の骨格

### 何を証明するか

Phase 0.5で証明すべき仮説は**1つだけ**:

> **「URLを入れるだけで出てくる診断結果が、ユーザーの次のアクション（依頼書作成・改善指示・チーム共有）を実際に変えるか」**

## 5. TO-BE設計

### 5.1 市場参入順序

| 順序 | 市場 | 理由 | 時期 |
|------|------|------|------|
| **1st（初手）** | 藤野型（代理業・フリーランス） | 即決可能、Tab 1で完結、自然バイラル（クライアントに送る）、品質フィードバックが明確 | Phase 0.5-1 |
| **2nd（本命）** | 浦田型（事業会社マーケ） | 最大市場、月額課金の安定性、GSC/GA4連携で粘着度↑ | Phase 1-2 |
| **3rd（拡張）** | 七條型（マーケ責任者） | チーム契約でARPU↑、但しTeam機能が必要 | Phase 2-3 |
| **後回し** | エンタープライズ（100名+マーケ部門） | PMF確立後。カスタマイズ要求が重い | Phase 3以降 |
| **捨てる** | 個人ブロガー・アフィリエイター | WTP低い、サポートコスト高い、プロダクト方向性と不整合 | 対象外 |

**なぜこの順序か:**

1. **藤野型が初手な理由**: 意思決定が速い（承認プロセスなし）、Tab 1（LP分析+依頼書）だけで「今日から使える」価値がある、クライアントLPを分析→結果URLを送る→クライアントが「自分も使いたい」＝自然なバイラルループ。最小機能で最大の学習が得られる。

2. **浦田型が本命な理由**: 市場規模が最大（日本のBtoB企業のマーケ部門）、月額課金の継続性が高い（一度承認されれば解約しにくい）、GSC/GA4連携でデータ粘着度が上がる。ただし初手にしない理由は「上長承認に時間がかかる」「Tab 1だけでは価値が薄い（自社データ連携がないと浅い）」。

3. **七條型が拡張な理由**: チーム機能（マルチユーザー、権限管理）が必須。Phase 0.5-1では作らない。ただしARPUが高いため、PMF確立後は最優先で攻める。

### 5.2 プロダクト定義

**一文定義:**
> Publish Gateは、URLを入れるだけで「なぜこのページの数字が悪いのか」を診断し、「誰に何をどう依頼すればいいか」まで出力する、マーケティング業務の診断→依頼代行ツールである。

**診断ツールとしての定義:**
- 入力: URL 1つ
- 処理: 企業理解 → ページ読解（Vision API + DOM） → 課題診断 → 依頼書生成
- 出力: 優先度付き課題リスト + 各課題のデザイナー/エンジニア向け依頼書
- 特徴: コピー文言は出さない（構造変化を図示）、薬機法・景表法チェック内蔵

**業務代替ツールとしての定義（Phase 1以降）:**
- 代替する業務: LP分析 → 課題整理 → 施策立案 → 依頼書作成 → 結果共有
- 代替しない業務: 実装そのもの、A/Bテスト実行、広告運用、SEO実装

### 5.3 Phase 0.5の役割（再定義）

**Phase 0.5は製品リリースではない。1仮説の検証装置である。**

検証する仮説:
> 「AIが生成したLP診断結果が、プロのマーケター（藤野型）の次のアクションを変えるか」

具体的には:
- 分析結果を見て「これは正しい」と感じるか（品質）
- 分析結果を見て「これを使おう」と思うか（行動変容）
- 分析結果を他者に見せたいと思うか（共有意欲）

**Phase 0.5に含めるもの:**
- Webダッシュボード（Tab 1のみ、タブUIなし）
- URL入力 → 分析結果表示 → 課題カード → 依頼パネル
- 共有URL生成（シンプルなリンクコピー）
- PoweredByバッジ

**Phase 0.5から除外するもの:**
- Chrome拡張（Phase 1へ）
- タブ2-6のロック表示（存在を示さない）
- ソーシャルシェアボタン（Twitter/Facebook）
- アカウント機能（Phase 1へ）
- Supabase統合（Phase 0.5ではインメモリ/KV）
- メール通知
- CSV出力

### 5.4 MVPのコアフロー

```
1. ユーザーがURLを入力（Webダッシュボード）
2. ローディング表示（4ステップ進捗）
3. 分析結果表示:
   a. 改善ポテンシャル（+XX%）
   b. 企業理解サマリー
   c. ページ読解結果
   d. 優先度順の課題リスト（アコーディオン）
   e. 各課題の依頼書（展開表示）
4. 共有URLコピーボタン
5. PoweredByバッジ
```

### 5.5 LP / Webダッシュボード / Chrome拡張の正しい責務分離

| 要素 | 責務 | Phase |
|------|------|-------|
| **LP（ランディングページ）** | 認知→Free登録の転換。プロダクトの価値を30秒で伝える。サンプル分析結果を見せる。 | Phase 1（Phase 0.5はβ直接配布のためLP不要） |
| **Webダッシュボード** | コア体験の全て。URL入力→分析→結果→依頼書→共有。全ユーザーの主要接点。 | Phase 0.5〜 |
| **Chrome拡張** | リテンション装置。「LPを見ているその瞬間」に分析を起動できる。Webダッシュボードへの導線。 | Phase 1〜 |

### 5.6 KPIの優先順位（Phase 0.5）

| 優先度 | 指標 | 定義 | 目標 |
|--------|------|------|------|
| **P0** | 診断品質スコア | β参加者が分析結果を「実際に使える」と評価した割合 | 3/5名以上 |
| **P1** | 行動変容率 | 分析結果を見た後に何らかのアクション（依頼書コピー、改善着手、チーム共有）を取った割合 | 3/5名以上 |
| **P2** | 2回目分析率 | 1回目の分析後、2週間以内に2回目の分析を行った割合 | 3/5名以上 |
| **P3** | 共有URL生成率 | 分析結果の共有URLを生成した割合 | 2/5名以上 |
| **P4** | NPS | 推奨度スコア | NPS 40+ |

**旧KPIとの差分**: 共有URL生成率をP0からP3に降格。診断品質と行動変容をP0-P1に昇格。

### 5.7 課金の成立条件

Starterへの課金が成立する条件:
1. Free枠（月5回）を使い切るユーザーが存在する（需要の証明）
2. 分析結果が「自分でやるより速い」と認識されている（品質の証明）
3. 依頼書が「実際にデザイナー/エンジニアに送った」実績がある（業務組み込みの証明）
4. 月5回では足りないと感じるユーザーが存在する（頻度の証明）

### 5.8 継続利用の成立条件

1. 分析品質が「毎回新しい発見がある」レベルを維持
2. 依頼書が「そのまま使える」粒度を維持
3. 分析対象URLが継続的に発生する業務フロー（代理業の新規案件、事業会社の定期LP改善）
4. 過去の分析結果を参照できる（Phase 1以降）

### 5.9 共有が起きる条件

1. 分析結果が「自分だけで持っていてはもったいない」品質
2. 共有先が明確（藤野型→クライアント、浦田型→上司/チーム、七條型→部下）
3. 共有URLが「自分の仕事の品質を上げて見せる」ツールとして機能する
4. 共有アクションが1クリック以内

### 5.10 失敗条件

以下のいずれかが発生した場合、Phase 0.5はNO-GO:
1. 5名中3名以上が「これならClaude/GPTに直接聞くのと同じ」と評価
2. 5名中3名以上が分析結果を見て「何もアクションを取らなかった」
3. 5名中3名以上が2回目の分析を行わなかった
4. 分析結果の精度が「自分の目で見たほうが速い」と評価された

### 5.11 将来拡張の正しい順序

```
Phase 0.5 (W1-4): Tab 1（LP分析+依頼書）+ 共有URL [Webのみ]
  ↓ GO判定
Phase 1 (W5-8): Chrome拡張 + アカウント + Supabase + Tab 2（広告訴求）+ Starter課金
  ↓ Starter転換率 >5%
Phase 2 (W9-16): Tab 3-4（市場/流入分析）+ GSC/GA4連携 + Pro課金
  ↓ Pro転換率 >10%
Phase 3 (W17-24): Tab 5-6（競合/事業分析）+ チーム機能 + Business課金
  ↓ Business契約 3社+
Phase 4 (W25+): API/MCP + ホワイトラベル + エンタープライズ
```

## 11. 最終提言

### Publish Gateが勝てる構造とは何か

Publish Gateの本質的な価値は「LP分析」ではない。**「マーケターが毎週やっている『分析→課題整理→依頼書作成』という業務サイクルの、最も苦痛な部分を30秒で終わらせること」**にある。

この認識に基づくと、以下の3点が戦略の骨格になる:

1. **初手は「診断品質」で勝つ**: 藤野型が「これは自分でやるより速いし、同等以上だ」と感じる品質がなければ、共有もバイラルも起きない。Phase 0.5の全リソースを診断品質に集中すべき。Chrome拡張やソーシャルシェアは品質が証明された後でいい。

2. **依頼書（Brief）が本当の差別化**: Claude/GPTに直接聞いても「分析結果っぽいもの」は出る。しかし「デザイナーに明日渡せる依頼書」は出ない。ここがPublish Gateの**模倣困難性の核**。依頼書の品質を磨き込むことが、「ChatGPTに聞けばいいじゃん」との差別化になる。

3. **バイラルは品質の結果であって手段ではない**: 共有URLは重要だが、「共有を促すUI」に最適化するのではなく、「共有したくなる品質」に最適化すべき。品質が高ければ共有は自然に起きる（藤野型はクライアントに送りたいから）。品質が低ければ、どんなに共有ボタンを目立たせても使われない。

### 最も重要なリスク

Phase 0.5の最大リスクは「AI分析の品質が期待を下回ること」である。藤野型は「プロのマーケター」であり、自分より下のレベルの分析には価値を感じない。Day 0テスト（β前の実LP品質検証）を必ず実施し、品質が基準に達しない場合はプロンプト改善に時間を使うべきであり、機能追加に走るべきではない。

### 捨てる勇気

Phase 0.5で最も重要なのは「何を作るか」ではなく「何を作らないか」である。Chrome拡張、タブ2-6、アカウント機能、ソーシャルシェア——これらは全て「あったほうがいい」機能だが、Phase 0.5の仮説検証には不要。1つの仮説を1つの装置で検証する。それがPhase 0.5の役割である。
