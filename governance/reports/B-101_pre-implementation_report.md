### 3. Decision Required

#### DR-5: 検証環境（止めるべき）

検証を実行する環境の選択:

- A) ローカル開発環境（`npm run dev`）から直接APIを叩く → 最も簡単。ただしネットワーク・API課金が発生
- B) デプロイ済みステージング環境 → β配布準備（B-104）の前提が必要
- C) テストスクリプトからAPIを直接叩く（E2Eではなく、lib関数を直接呼ぶ） → 最も制御しやすい

**推奨: C** — `runAnalysis(url)` を直接呼び出すスクリプトが最も再現性が高く、結果をJSONで保存しやすい。UI層の不確実性を排除できる。

#### DR-6: 20サイトの選定方針（確認が望ましい）

- A) 全20サイトを人間が指定する → 最も精度が高いが、時間がかかる
- B) 業種・構造パターンの分布基準をClaude Codeが設計し、具体サイトは人間が指定 → 分業
- C) TEST-SCENARIOS.jsonの10パターンを基盤に、各パターンにつき実サイト2つを人間が指定（計20サイト） → 既存資産を活用

**推奨: C** — TEST-SCENARIOS.jsonが既に業種・訴求タイプ・法規制リスクの分布を網羅している。各パターンに実サイトを2つ割り当てることで、パターン網羅性と現実性を両立。

#### DR-7: Vision API（スクリーンショット）の検証有無（確認が望ましい）

- A) Vision API あり/なし の両方で検証する → 精度差が測定できる。コスト2倍
- B) Vision API ありのみ → 本番と同条件。ただしSCREENSHOT_API_KEY必要
- C) Vision API なしのみ → 最も安価。ただしCLAUDE.md「Vision APIは省略しない」に反する

**推奨: B（可能なら）、不可ならC** — SCREENSHOT_API_KEYの利用可否による。

#### DR-8: 合格ライン「80%」の解釈（止めるべき）

backlog B-101の受け入れ条件は「80%以上のスコア」。この「80%」が何を指すか:

- A) 20サイト中16サイト以上で「合格」判定（サイト単位の合否率）
- B) 全評価項目の加重平均スコアが80点/100点以上（スコア平均）
- C) 重要項目（信頼感・課題特定の妥当性）で80%以上、かつ致命的失敗がゼロ

**推奨: C** — 平均で80%を超えても、薬機法の見逃しが1件あれば「信頼感」は崩壊する。重要項目の基準達成 + 致命的失敗ゼロの組み合わせが適切。

#### 4-1. 20サイトの選定方法

| 3 | S03: コンサル/制作会社 | サービス | ブランド訴求 | なし | 2 |

#### 4-2. 評価観点

5つの評価軸。優先順位に対応（信頼感 > 分析精度 > 速さ > 依頼文品質）。

| 評価軸 | 重み | 評価内容 | 優先順位対応 |
|--------|------|---------|------------|
| A. 課題特定の妥当性 | 30% | 指摘された課題が実際にそのLPに存在するか。的外れな指摘がないか | 信頼感 + 分析精度 |
| B. 見逃しの少なさ | 25% | 明らかな課題が指摘されていないケースがないか | 分析精度 |
| C. 法規制検知の正確性 | 20% | 薬機法・景表法リスクの検知が正確か（偽陽性・偽陰性） | 信頼感 |
| D. 依頼文の行動可能性 | 15% | デザイナー/エンジニアが依頼文だけで作業に着手できるか | 依頼文品質 |
| E. パイプラインの安定性 | 10% | エラーなく完了するか。30秒以内か。JSON構造は正しいか | 速さ |

#### 4-3. スコアリング方法

各評価軸を5段階（1〜5）で採点。加重平均で総合スコアを算出。

| スコア | 意味 | 判断基準 |
|--------|------|---------|
| 5 | 優秀 | プロのマーケコンサルと同等以上の指摘 |
| 4 | 良好 | 主要な課題を漏れなく特定。軽微な改善余地あり |
| 3 | 合格 | 主要課題は特定できているが、深さや具体性に改善余地 |
| 2 | 不足 | 重要な課題の見逃しまたは的外れな指摘がある |
| 1 | 不合格 | 致命的な誤りまたは出力が使い物にならない |

**総合スコア計算:**

```
総合 = A×0.30 + B×0.25 + C×0.20 + D×0.15 + E×0.10
```

100点満点換算: 総合スコア × 20

#### 4-4. 合格ライン

| 条件 | 基準 | 根拠 |
|------|------|------|
| 総合スコア平均 | 80点/100点以上（加重平均4.0/5.0以上） | B-101受け入れ条件 |
| 致命的失敗ゼロ | 全20サイトで「スコア1」がゼロ | 信頼感最優先 |
| 法規制検知 | 法規制リスクがあるサイト（S02,S07,S09,S10対応の8サイト）で検知漏れゼロ | 信頼感 |
| パイプライン完走率 | 20サイト中19サイト以上（95%）でエラーなく完了 | 安定性 |

#### 4-5. 依頼文品質の評価方法

依頼文（BriefPanel）を以下の観点で評価:

| 観点 | 5点（優秀） | 3点（合格） | 1点（不合格） |
|------|-----------|-----------|-------------|
| 目的の明確さ | 何を改善するか一読で分かる | 改善方向は分かるが曖昧さがある | 何を改善したいのか不明 |
| 方向性の具体性 | デザイナーが迷わず着手できる | 方向は分かるが解釈の余地がある | 抽象的すぎて着手不能 |
| 制約条件の妥当性 | ブランドトーンや技術制約を踏まえている | 一般的な制約のみ | 制約が的外れまたは欠落 |
| QAチェックリスト | 完了判定に使える具体的な項目 | 項目はあるが検証しにくい | 形骸化している |

#### 4-6. エラー・失敗パターンの分類

| カテゴリ | コード | 説明 | 重大度 |
|---------|--------|------|--------|
| パイプラインエラー | E-PIPE | API呼び出し失敗、タイムアウト、JSON parse失敗 | 高 |
| 的外れ指摘 | E-IRRELEVANT | そのLPに存在しない課題を指摘 | 高 |
| 重要課題見逃し | E-MISS | 明らかな課題（FVの致命的問題等）を指摘しない | 高 |
| 法規制見逃し | E-REG-MISS | 薬機法/景表法リスクがあるのに検知しない | 致命的 |
| 法規制誤検知 | E-REG-FALSE | リスクがないのに検知する | 中 |
| 依頼文不備 | E-BRIEF | 依頼文がデザイナー/エンジニアに伝わらない | 中 |
| 構造エラー | E-STRUCT | 出力JSONが型定義と不一致 | 高 |
| 速度超過 | E-SLOW | 30秒以内に完了しない | 中 |
| 過剰指摘 | E-OVER | 高品質LPに対して不要な改善を多数指摘 | 中 |
| 企業理解ミス | E-COMPANY | 業種・ビジネスモデルの誤認識 | 高 |

#### 評価観点（再掲）

A. 課題特定の妥当性（30%）
B. 見逃しの少なさ（25%）
C. 法規制検知の正確性（20%）
D. 依頼文の行動可能性（15%）
E. パイプラインの安定性（10%）

#### スコアリング方法

- 各軸5段階（1〜5）
- 加重平均 × 20 = 100点満点
- 法規制チェック不要のサイトはC軸を除外し、残り4軸で重み再配分（A:37.5%, B:31.25%, D:18.75%, E:12.5%）

#### 合格ライン

- 加重平均80点以上
- 致命的失敗（スコア1）ゼロ
- 法規制検知漏れゼロ（対象8サイト中）
- パイプライン完走率95%以上

#### 出力サンプルの見方

検証実行時、各サイトの分析結果JSON全体を保存し、以下の順で読む:

1. `pipeline_status` → 完走したか
2. `issues` → 指摘数と内容の妥当性
3. `regulatory` → 法規制リスクの検知有無
4. `issues[].brief` → 依頼文の具体性
5. `metadata.analysis_duration_ms` → 速度

#### 失敗パターン分類（再掲）

致命的: E-REG-MISS（法規制見逃し）
高: E-PIPE, E-IRRELEVANT, E-MISS, E-STRUCT, E-COMPANY
中: E-REG-FALSE, E-BRIEF, E-SLOW, E-OVER

#### 検証結果の記録先

- サイト別結果JSON: 検証実行時に決定（governance/またはテスト用ディレクトリ）
- 全体サマリー: `governance/02_DECISION_LOG.md` にB-101の検証結果として記録
