# 02_DECISION_LOG.md — 判断ログ

> 実装中に発生した判断を記録する。
> 人間が判断したもの、Claude Code が判断を仰いだもの、両方を残す。
> 「なぜそうなったか」を後から追跡できることが目的。

---

## 記録フォーマット

```
### DEC-XXX: [判断タイトル]
- **日付**: YYYY-MM-DD
- **起票者**: 人間 / Claude Code
- **判断者**: 人間 / Claude Code（許可範囲内）
- **論点**: 何を決める必要があったか
- **選択肢**: A / B / C
- **決定**: 選んだもの
- **理由**: なぜそれを選んだか
- **影響範囲**: 何に影響するか
- **関連バックログ**: B-XXX
```

## 判断記録

### DEC-001: ガバナンスファイルの配置場所
- **日付**: 2026-03-14
- **起票者**: Claude Code
- **判断者**: 人間（タスク指示内で承認）
- **論点**: ガバナンスファイルを `.governance/`（dotディレクトリ）vs `governance/`（通常ディレクトリ）のどちらに置くか
- **選択肢**: A) `.governance/` / B) `governance/`
- **決定**: B) `governance/`
- **理由**: dotディレクトリはツールが無視しがち。視認性と参照しやすさを優先
- **影響範囲**: 全ガバナンスファイルのパス
- **関連バックログ**: なし（ガバナンス基盤整備）

### DEC-002: Phase 0.5 必須体験フローの線引き
- **日付**: 2026-03-15
- **起票者**: Claude Code
- **判断者**: 人間
- **論点**: Phase 0.5 の正式体験として定義された9ステップ（URL入力→課題構造→重要Issue→施策案→バックログ→依頼文→共有→再訪導線→有料差分認知）のうち、どこまでを「必須」とするか
- **選択肢**:
  - A) 9ステップ全てを必須 — スコープ拡大（バックログ表示・有料訴求UIの追加が必要）
  - B) コア5ステップ（URL入力→課題構造→重要Issue→依頼文→共有）を必須、残り4つを「あれば強い」 — 最小スコープでセンターピン直撃
  - C) 現在の実装をそのまま正式体験と定義 — 最速だが体験定義が曖昧なまま
- **決定**: B
- **理由**: センターピンは「藤野型が結果URLをクライアントに送る」。必要十分なのは「課題が見える→依頼文がある→共有できる」の流れ。施策案独立表示・バックログ表示・再訪導線強化・有料差分訴求は「あれば強い」に格下げ
- **影響範囲**: `governance/10_ARCHITECTURE_BOUNDARY.md` に体験定義を追記。`governance/01_PHASE_0_5_BACKLOG.md` に必須/あれば強い/不要の対応表を追記
- **関連バックログ**: B-001〜B-006（全て DONE で必須体験をカバー）

### DEC-003: 「課題構造」と「重要Issue」の定義
- **日付**: 2026-03-15
- **起票者**: Claude Code
- **判断者**: 人間
- **論点**: 現在の実装では `issues` 配列がフラットに並んでいる。「課題構造→重要Issue」の段階表現をどう解釈するか
- **選択肢**:
  - A) 課題構造 = 企業理解+ページ読取+改善ポテンシャル、重要Issue = priority上位のissue — 現在の実装で対応済み
  - B) 課題構造 = issueのカテゴリ分類ツリー、重要Issue = 最重要ノード — 新規UIが必要
- **決定**: A
- **理由**: 現在の出力JSON（`company_understanding` + `page_reading` = 課題構造、`issues` のpriority上位 = 重要Issue）で体験として成立。新しい情報構造UIの追加は不要
- **影響範囲**: `governance/10_ARCHITECTURE_BOUNDARY.md` の体験定義に用語対応を明記
- **関連バックログ**: B-002

### DEC-004: 体験定義・責務分離の文書化先
- **日付**: 2026-03-15
- **起票者**: Claude Code
- **判断者**: 人間
- **論点**: Phase 0.5 の体験定義と責務分離をどこに文書化するか
- **選択肢**:
  - A) `governance/10_ARCHITECTURE_BOUNDARY.md` に追記 — 既存ファイルの拡張。コンパクト
  - B) 新規 `governance/11_EXPERIENCE_DEFINITION.md` を作成 — 体験定義として独立
  - C) `01_PHASE_0_5_BACKLOG.md` の冒頭に追加 — バックログと一体管理
- **決定**: A
- **理由**: 新規ファイルを増やさず、アーキテクチャ境界の延長として体験定義と責務分離を一元管理。バックログからはリンクで参照
- **影響範囲**: `governance/10_ARCHITECTURE_BOUNDARY.md` に2セクション追加。`governance/01_PHASE_0_5_BACKLOG.md` に参照リンクと必須/あれば強い/不要の対応表を追加
- **関連バックログ**: B-001〜B-006
